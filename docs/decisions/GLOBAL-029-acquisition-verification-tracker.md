# GLOBAL-029 — `docs/research/automated-icp-validation-plan-verification.md` is the canonical user-flow verification tracker; it mirrors the impl plan one-for-one; both files are agent-ran and both are exempt from the 20 KB cap

- **Decision:** `docs/research/automated-icp-validation-plan-verification.md` is the single source of truth for whether each user-flow listed in [`docs/research/automated-icp-validation-plan.md`](../research/automated-icp-validation-plan.md) §8 has been **walked end-to-end by an agent against a real surface**. Every `FLOW-NNN` appears in BOTH files with the same ID: the impl plan tracks *what is shipped* (sub-tasks, SK-* refs, % implemented); the verification plan tracks *what has been verified* (walkthrough steps, required credentials, last-verified date, outcome log). Both files are agent-ran — adding, modifying, or superseding a `FLOW-NNN` updates BOTH in lockstep, in the same PR. Both files are **exempt from the 20 KB cap** set by [`CLAUDE.md §2 P4 D4`](../../CLAUDE.md) (the D4 bullet holds the canonical exemption list); the [`GLOBAL-028`](./GLOBAL-028-acquisition-progress-tracker.md) exemption is extended to the mirror file for the same reason it applies to the impl plan — evidence accumulates over time, hard truncation would drop the data that proves what failed.

- **Core value:** Goal-first, Bullet-proof, Honest latency

- **Why:** Implementation and verification are independently meaningful axes. A flow can be "fully implemented" (every sub-task ticked, every SK-* shipped) and still be broken in the user-visible composition — that's the §1.1 stranger-test gap. A flow can be "verified" by founder demo and still skip past a documented SK decision — that's a stale demo. Keeping the two trackers separate and ID-aligned makes the gap visible: the dashboard view is "implementation % vs verification %". One file with both axes would push the size + cognitive load past the practical bar; per-flow files under `flows/<id>/` would fragment the dashboard. The mirror-file shape solves both. The "agent-ran" requirement is what keeps verification honest: the verification file *is* the walkthrough script — an agent reads it, performs the steps, writes outcome back; there is no human-only verification path that escapes the audit trail.

- **Consequence in code:** Adding a new flow = one block in `automated-icp-validation-plan.md` §8 (with sub-tasks + SK-* refs + `Mirror: see verification.md FLOW-NNN`) AND one block in `automated-icp-validation-plan-verification.md` (with walkthrough steps + required credentials + outcome log + `Mirror: see automated-icp-validation-plan.md §8 FLOW-NNN`) in the same PR. Modifying a flow's implementation sub-tasks does NOT require re-verifying — the verification plan tracks the agent walk; the impl plan tracks the SK-* land. The `CLAUDE.md` §5 path-map glob `docs/research/automated-icp-validation-plan*.md` covers both files via one entry. A `## Status` block at the top of each file is the dashboard view: implementation % per flow (impl plan) and verification % per flow (verification plan). Updates to either status block happen in the same PR that lands the underlying change.

- **Alternatives rejected:**
  - **Verification as a subsection inside the impl plan.** Couples the two axes; the verification walkthrough scripts wouldn't fit cleanly under impl sub-tasks; rotating one without the other creates silent drift; the file would cross 20 KB faster than necessary even with the exemption.
  - **Single file with both axes interleaved (e.g. `## Implementation` / `## Verification` per flow).** Same coupling problem as the subsection alternative, plus the dashboard view ("aggregate impl % vs verify %") becomes impossible without scripted aggregation.
  - **Per-flow files under `docs/research/flows/<id>/{impl,verify}.md`.** Fragmentation — the dashboard requires reading 7+ files, the cross-flow comparison view dies. The mirror-file shape gives one-page-per-axis dashboards plus per-flow blocks.
  - **A typed schema + script (TypeScript / YAML).** Over-engineering for v1: the verification file's value is that an agent *reads* it as a walkthrough; markdown is the right shape. A typed schema becomes valuable once ≥15 flows exist and aggregation queries land.
