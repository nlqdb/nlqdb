# GLOBAL-032 — The five canonical user flows in the acquisition tracker; each must have at least one agent-runnable end-to-end walker that ran against the deployed surface within the last seven days

- **Decision:** Of the `FLOW-NNN` blocks in the acquisition tracker, the five that cover the inbound funnel — **FLOW-001** (anonymous-first happy path, P1), **FLOW-002** (`/solve/<slug>` AEO inbound, P3), **FLOW-003** (`/vs/<competitor>` comparison inbound, P3/P4), **FLOW-004** (waitlist → invite → gate bypass, P1 invited), and **FLOW-005** (agent self-provisions DB via MCP, P2) — are the canonical "five most important user flows nlqdb provides". Every other `FLOW-NNN` is either post-acquisition (FLOW-006 SDK escape hatch, FLOW-007 anonymous-DB adoption) or a system flow (FLOW-008 cron source-health). Each of the five MUST have at least one bash/HTTP/Playwright walker checked into `scripts/` or `tools/stranger-test/` that an agent can run with zero pre-provisioned credentials beyond what's already in `verify-secrets.sh`, that walks the deployed surface end-to-end (not a local dev server unless the walker explicitly says so), and that was last run successfully within the last seven days. The walker run is recorded in the verification mirror's per-flow `Outcome log` table with a UTC date, agent identity, state (`passed` / `failed step N` / `blocked credentials` / `blocked upstream` / `inconclusive`), and one sentence of notes. The impl plan's §0.5 dashboard surfaces the canonical-five with their most recent verification date. A walker that hasn't run in the last seven days is treated as the next agent's highest-priority pick — equivalent to a stranger-test regression — because acquisition tracking cannot honestly claim "this funnel works today" from stale evidence.

- **Core value:** Bullet-proof, Honest latency, Goal-first

- **Why:** The acquisition tracker has accumulated eight `FLOW-NNN` blocks across personas P1/P2/P3/P4/cron, but agents reading the dashboard have had no formal answer to "which flows are the load-bearing five we cannot ship broken." Without that pin, every "what to pick next" answer is a judgement call that drifts. The canonical-five set is anchored in the persona priority encoded by [`personas.md`](../research/personas.md) (P1 highest, P2/P3 strong, P4 third-priority) and by the stranger-test seeded-prompt split (P1×10, P2×8, P3×4, P6×3 per [`tools/stranger-test/src/personas.ts`](../../tools/stranger-test/src/personas.ts)) and by the ICP-mining cron's persona-fit rubric (`runIcpScore` in [`apps/api/src/icp-score.ts`](../../apps/api/src/icp-score.ts) scores P1/P2/P3/P6 — the same four personas the canonical-five cover plus P6 which currently lacks a dedicated flow and is the natural next gap). The seven-day freshness rule prevents the "shipped, never re-walked" failure mode the §1.1 stranger-test gap was named to detect at scale: a walker that exists but hasn't run is exactly as useful as no walker, because the surface under test could have regressed silently the day after the last walk.

- **Consequence in code:** (1) The impl plan [`automated-icp-validation-plan.md`](../research/automated-icp-validation-plan.md) carries a `§0.5 The five user flows that matter most` section at the top with a pinned table naming the five, the persona each serves, the canonical walker, and the last-verified date. (2) The verification mirror [`automated-icp-validation-plan-verification.md`](../research/automated-icp-validation-plan-verification.md) carries the same table in its status dashboard. (3) Every PR touching the acquisition trackers refreshes the last-verified column for any of the five whose walker it re-ran; agents whose PR makes no acquisition-surface change still re-run the canonical-five walkers as part of the GLOBAL-030 self-review. (4) Per-flow walkers: FLOW-001/002/003 → `scripts/stranger-test.sh` (+ `scripts/stranger-test-invited.sh` for the invite-bearing variant); FLOW-004 → `scripts/flow-004-walk.sh`; FLOW-005 → `scripts/flow-005-walk.sh` (added by the PR landing this GLOBAL); plus the static-subset overlay `scripts/verify-flows.sh` that covers the curl-observable steps of all five. (5) The daily `.github/workflows/acquisition-health.yml` cron runs every walker daily at 06:00 UTC so the seven-day freshness rule is met by default — the next agent reads the artifact via `mcp__github__list_workflow_runs`. (6) Adding a sixth canonical flow requires editing this decision; demoting one of the five requires the same. The set is small on purpose — "five" is the bar above which the dashboard stops being a dashboard.

- **Alternatives rejected:**
  - **No canonical-five pin; let the §8 list speak for itself.** That's the status quo, and it has shipped exactly one fully-verified flow (FLOW-004) across two months because no one was forced to answer "is the inbound funnel green today?" — they could always point at a partially-verified flow and call the funnel "mostly working."
  - **Canonical-three (FLOW-001/004/005) — only the flows that hit `/v1/ask`.** Drops the AEO + comparison surfaces (`/solve/`, `/vs/`), which are the only top-of-funnel SEO traps; the inbound funnel is broken if those don't deliver. Three doesn't cover P3/P4.
  - **Canonical-seven (every user flow including FLOW-006/007).** Mixes acquisition with retention/escalation. FLOW-006 is the SDK escape hatch — by the time a user invokes `runSql` they're already through acquisition. FLOW-007 is the anonymous→authed adoption boundary — same. Putting them in the same dashboard as FLOW-001 dilutes the signal "is acquisition working today" with "is adoption working today" — two different questions.
  - **A typed schema (TypeScript / YAML) listing the canonical-five with a script that validates per-PR.** Over-engineered while the set is five — the GLOBAL plus the §0.5 table plus the mirror dashboard are enough audit surface. Promote to a typed schema once the set grows past ten or once aggregation queries land.
  - **No seven-day freshness rule.** Without it, the `passed YYYY-MM-DD` line in the verification mirror is unbounded in age — a walker that hasn't run since 2025 still counts as "passed." That's exactly the "shipped, never re-walked" trap this GLOBAL exists to close.
  - **Hard cron failure (exit non-zero on stale walker) instead of soft "next agent's priority #1."** `acquisition-health.yml` exits 0 unconditionally per [`SK-STRG-003`](../features/stranger-test/FEATURE.md) to keep the operator loop intact (no founder-facing notification channel — the operator runs the agent prompt and the agent re-walks; that's the loop). Hard failure on staleness would re-introduce the email channel the operator-loop forbids.
