# SK-HDC-015 — Compiler auto-generates defaults for single-column integer/uuid primary keys

- **Decision:** When the `SchemaPlan` has a single-column primary key of type `integer`, `bigint`, or `uuid` and no explicit `default`, `apps/api/src/db-create/compile-ddl.ts:compileColumn` appends `GENERATED BY DEFAULT AS IDENTITY` (int family) or `DEFAULT gen_random_uuid()` (uuid). Composite primary keys, `text` PKs, and columns that already carry an LLM-supplied `default` are untouched.
- **Core value:** Bullet-proof, Seamless auth
- **Why:** The schema-inference prompt (`packages/llm/src/prompts/schema-inference.ts`) does not teach the LLM about `IDENTITY` / `SERIAL` / `gen_random_uuid()`; it almost always emits `id` as `{type: integer, nullable: false, default: null}`. The planner LLM then emits `INSERT INTO t (name, ...) VALUES (...)` without `id`, Postgres rejects the row on the NOT NULL primary key, the exec catch in `apps/api/src/ask/orchestrate.ts:377` maps that to the generic `db_unreachable` envelope, and the chat renders `Couldn't reach the database — try again.` — i.e. the seamless `anon-create → sign-in → write` arc that `SK-ANON-014` is supposed to guarantee dead-ends at the first authed write. Reproduced live on the PR-192 preview Worker (anon `create users table` → mock sign-in → `add row: name='Alice', email='alice@example.com'` → 502 `db_unreachable`, planner-emitted SQL `INSERT INTO ...users (name, email) VALUES (...)`). Putting the default policy in the deterministic compiler per `SK-HDC-002` keeps both LLM calls naive about PK shape (schema-inference doesn't need to know IDENTITY syntax; the planner doesn't need to reason about which columns to include in INSERT lists). `BY DEFAULT` rather than `ALWAYS` so `apps/api/src/db-create/neon-provision.ts`'s sample-row inserts can still pass explicit ids without being rejected.
- **Consequence in code:** `compileTable` builds `isSingleAutoPk` from `primary_key.length === 1`; `compileColumn` accepts the flag and picks among explicit-`default` → `GENERATED BY DEFAULT AS IDENTITY` → `DEFAULT gen_random_uuid()` → no default. `compile-ddl.test.ts` updated for the UUID-PK case. Sample-row INSERTs in `neon-provision.ts` are unchanged (explicit ids still accepted under `BY DEFAULT`). The fix is centralised in the DDL compiler per `SK-HDC-002` — every surface that creates a database (`@nlqdb/sdk` create, `nlq` CLI, `@nlqdb/mcp` server, `<nlq-data>` element) goes through `/v1/ask`'s create path, so all surfaces inherit the behaviour via the API per `GLOBAL-003` without surface-side changes. Surface feature docs that describe table-create response shape don't need to repeat the rule; reference `SK-HDC-015` if PK-default behaviour is called out explicitly.
- **Alternatives rejected:**
  - **Update the schema-inference prompt to emit `default: "gen_random_uuid()"` on PK columns.** LLM non-determinism produces partial coverage, hallucinated function names, and quoting bugs; the deterministic compiler is `SK-HDC-002`'s seam.
  - **Update the planner LLM to include all NOT NULL columns in INSERTs.** Pushes complexity into the read/write path and asks the LLM to invent values for arbitrary types (what's a sensible default for `created_at`? `status`?).
  - **`GENERATED ALWAYS AS IDENTITY` instead of `BY DEFAULT`.** Forbids explicit ids in `INSERT`, breaking the provisioner's sample-row inserts (`neon-provision.ts` passes the LLM's `sample_rows[].values.id`).
  - **`SERIAL` / `uuid_generate_v4()`.** Pre-PG10 and extension-required forms; `IDENTITY` + `gen_random_uuid()` are PG13+ built-ins and Neon runs PG16.
