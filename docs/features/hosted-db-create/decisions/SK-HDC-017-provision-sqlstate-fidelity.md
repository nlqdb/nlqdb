# SK-HDC-017 — Provisioner maps Postgres SQLSTATE *classes* and pins the raw SQLSTATE on the failure span

- **Decision:** `mapTransactionError` in `apps/api/src/db-create/neon-provision.ts` classifies a failed provision transaction by Postgres SQLSTATE **class**, not a four-code lookup: class `22` (data exception) and `23` (integrity violation) → `sample_insert_failed`; class `42` (syntax / undefined-object / access-rule) → `ddl_execution_failed`, except `42P06` (duplicate_schema) which stays `schema_already_exists` because it drives the orchestrator's id-collision retry; an error with **no** SQLSTATE (TLS reset, timeout) keeps the catch-all `transaction_failed`. The `db.transaction` span additionally carries `db.transaction.error_sqlstate` (the raw 5-char code, or `none`) on every failure. The `ProvisionFailureReason` union is unchanged — this only sharpens which bucket each real failure lands in.
- **Core value:** Honest latency, Bullet-proof, Goal-first
- **Why:** The hosted-DB provision is the last leg of the only acquisition path that crosses the pre-alpha gate (FLOW-004: waitlist → invite → `/v1/ask` → auto-provision). On 2026-06-06 a FLOW-004 walk regressed to a hard `HTTP 500 {kind:"provision_failed",reason:"transaction_failed"}` — and the operator-loop agent (whose only tool is the deployed-surface HTTP response, no Grafana) could not tell whether the funnel broke on **engine quality** (the free chain emitted DDL with a hallucinated type, SQLSTATE `42704`, or a sample value that doesn't parse, `22P02`) or on **infra** (connection reset, no SQLSTATE). The old four-code map collapsed both `42704` and `22P02` into the opaque `transaction_failed`, so the single most diagnostic byte — the SQLSTATE class — was thrown away exactly where the tracker most needs it: the `reason` now ties a broken funnel back to the engine-quality bottleneck the whole acquisition plan is gated on, and the span attribute lets a log-based alert split "source dead" from "bad DDL" without a Tempo deep-dive.
- **Consequence in code:** `mapTransactionError` switches from per-code `if`s to class prefixes (`22`/`23` → `sample_insert_failed`, `42` → `ddl_execution_failed`, `42P06` first), a small `sqlStateOf(err)` helper extracts the code once for both the mapping and the span, and the transaction catch block sets `db.transaction.error_sqlstate`. Caveat: class 42 = "syntax error *or access-rule violation*", so the rare `42501` (insufficient_privilege — an infra failure on the prefix `CREATE ROLE`/`GRANT`) also lands in `ddl_execution_failed`; the `reason` bucket is a coarse phase hint, and the raw `error_sqlstate` span attribute is the field that tells a genuine engine-quality DDL error apart from a privilege/infra one. That is precisely why the span attribute ships alongside the coarser `reason`, rather than relying on the bucket alone. The attribute is cataloged in [`docs/performance.md`](../../../performance.md) §3 (`db.transaction` row) before it ships, per GLOBAL-014. Tests in `neon-provision.test.ts` pin `42704`/`42601` → `ddl_execution_failed`, `22P02` → `sample_insert_failed`, `42P06` → `schema_already_exists` (retry preserved), and the existing no-SQLSTATE → `transaction_failed` catch-all. No user-facing wire shape changes — the `provision_failed` envelope keeps GLOBAL-012's one-sentence contract; only the machine `reason` code and the span gain fidelity.
- **Alternatives rejected:**
  - **Surface the raw Postgres message in the HTTP response.** Violates GLOBAL-012 (one-sentence, no internals) and leaks schema/identifier detail to anonymous callers. The SQLSTATE belongs on the span, the *class* in the `reason`.
  - **Add new `ProvisionFailureReason` values (e.g. `syntax_error`, `data_error`).** Widens the union and every consumer's switch for no behavioural gain — the existing `ddl_execution_failed` / `sample_insert_failed` buckets already carry the phase signal; the span attribute carries the exact code for anyone who needs it.
  - **Auto-repair the DDL on `ddl_execution_failed` (retry with an LLM fix).** Out of scope and against P5 — the orchestrator deliberately only retries `schema_already_exists`. A repair loop is a quality-eval concern; this decision just makes the failure *legible* so that work can be prioritised on evidence.
  - **Leave the four-code map; read the SQLSTATE from Tempo when needed.** That is the status quo that cost a full reverse-engineering session (reproducing the transaction over Neon's HTTP endpoint) to learn one byte. The operator loop has no Grafana; the evidence must ride the surface the loop can read.
