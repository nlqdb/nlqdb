# SK-HDC-020 — A plan that fails Zod is re-inferred once with the validation issues fed back

- **Decision:** When `inferSchema`'s first `SchemaPlan` candidate fails Zod validation (`plan_invalid`), the module re-calls `deps.llm.schemaInfer` **once** with `previousAttempt = { plan, issues }` — the rejected plan (capped at 6 000 chars) plus a path-and-message summary of the Zod issues — before surfacing `plan_invalid` to the caller. `buildSchemaInferUser` appends a diagnostic-first correction block (the same shape as the planner's `SK-LLM-018` retry framing): keep everything that validated, fix only what the validator named, rename any reserved-word identifier. The repair is bounded to one extra attempt (`INFER_MAX_ATTEMPTS = 2`); a second failure returns `plan_invalid` exactly as before. The `llm_failed` (provider throw) and `ambiguous_goal` (shallow-plan) paths are unchanged — repair fires only on a deterministic Zod failure of an otherwise-complete plan. The Zod issue text is fed back into our own next prompt only, never returned to the client (`GLOBAL-012` — the client still sees only `issue_count`).
- **Core value:** Engine quality, Goal-first, Bullet-proof
- **Why:** The FLOW-004 seed-quality probe ([`SK-STRG-008`](../../stranger-test/decisions/SK-STRG-008-flow-004-seed-quality-probe.md)) measured 2026-06-17 that the seeding bottleneck SK-HDC-018/019 fixed has moved: on a harder 8-goal P1 set, `seeded_ok_ratio` is 1.0 over *classified creates*, but **3 of 8 goals hard-failed at provision with `infer_failed` (HTTP 422)** — the invited stranger got no DB at all, the worst first-value outcome. Captured against prod, the deterministic slice of those failures is `plan_invalid` with `issue_count: 1`: the model named a column a Postgres reserved word the schema-inference prompt's short example list (`select, table, user, order, group, …`) doesn't cover, so Zod rejects an otherwise-complete plan. Measured same-seed against the live `schema_infer` chain leader (Cerebras `gpt-oss-120b`, temp 0): the "a fitness log tracking workouts with sets and reps per exercise" goal infers **valid 0/3**; feeding the single Zod issue back recovers it **3/3**. Execution-guided repair is the highest-yield text-to-SQL technique ([`SK-ASK-022`](../../ask-pipeline/FEATURE.md#sk-ask-022) applies it to the planner); the typed-plan inference stage had no equivalent — a near-miss plan hard-failed instead of being nudged. The full PG reserved-word list (~100 words) is too large to enumerate in the prompt and would still miss the next class of one-issue rejections; a generic "fix what the validator named" repair covers reserved words and every other single-issue shape error without prompt bloat.
- **Consequence in code:** `apps/api/src/db-create/infer-schema.ts` wraps its single LLM-call-then-validate body in a 2-iteration loop; on a first Zod failure it stashes `previousAttempt` and continues, on the second it returns `plan_invalid`. `packages/llm/src/types.ts` adds `SchemaInferRequest.previousAttempt?: { plan: string; issues: string }`; `packages/llm/src/prompts/schema-inference.ts::buildSchemaInferUser` appends the correction block when it is set — every provider reuses that builder, so there is no per-provider plumbing (mirrors `PlanRequest.previousAttempt`). `infer-schema.test.ts` is the measured before/after: a reserved-word first plan + clean second plan returns `ok` after exactly two calls with `previousAttempt` carrying the rejected plan + the "reserved word" issue; a twice-invalid plan returns `plan_invalid` after two calls; the happy path stays one call. Failure-path only ⇒ zero added latency or cost on a create whose first plan already validates. The remaining `infer_failed` slice — transient `llm_failed` from chain exhaustion (the dead Gemini leg, [`SK-LLM-039`](../../llm-router/FEATURE.md)) — is provider-side and out of scope here.
- **Alternatives rejected:**
  - **Enumerate the full PG reserved-word list in the prompt.** ~100 words bloats every schema-infer prompt (token cost on the common success path) and still wouldn't catch the next non-reserved-word single-issue rejection. A generic validator-guided repair is smaller and covers the whole `plan_invalid` class.
  - **Deterministically auto-rename reserved-word identifiers in code.** Would have to rewrite the name in every back-reference (primary_key, foreign_keys, sample_rows, metrics, dimensions) and only ever fixes the reserved-word case, not other single-issue shapes. The one-call repair is simpler (P5) and general.
  - **Retry on `llm_failed` too.** That is a provider throw — the router chain already does provider failover (`SK-LLM-006`); a re-call with the same exhausted free chain re-throws. Repair is scoped to the deterministic Zod failure it actually fixes.
  - **No repair (the pre-2026-06-17 behaviour).** A near-complete plan with one reserved-word column deterministically hard-fails the create as `infer_failed` — the harshest first-value outcome, and the one the seed-quality probe now flags. Distinct from [`SK-HDC-019`](./SK-HDC-019-deterministic-sample-row-salvage.md)'s rejection of an LLM call to repair a *decorative sample row* (the DB still exists there); here the second call is the difference between a working database and none.
