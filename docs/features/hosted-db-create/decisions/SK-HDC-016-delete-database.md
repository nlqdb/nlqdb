# SK-HDC-016 — `DELETE /v1/databases/:id` reuses the `dropSchemaAndRegistry` rollback primitive; UI gates with typed-name confirmation

- **Decision:** A new authed endpoint `DELETE /v1/databases/:id` (`requireSession`-gated) calls `apps/api/src/db-create/neon-provision.ts`'s `dropSchemaAndRegistry(tracer, deps.pg, c.env.DB, dbId, schemaName)` — the existing SK-HDC-011 rollback primitive — then issues `DELETE FROM api_keys WHERE db_id = ?` to clean up orphan `pk_live_*` keys (SK-APIKEYS-001 binds those per-DB). The route returns `204 No Content` on success and `404 db_not_found` when the dbId is unknown to the caller's tenant (resolved via `resolveDb`, which is tenant-scoped — a cross-tenant id leaks no existence signal because both "wrong tenant" and "missing" map to 404). The web surface's chat side-panel (`apps/web/src/components/chat/LeftRail.tsx`) adds a per-row delete affordance that opens a confirmation dialog requiring the user to type the database's `displayName` exactly; the SDK exposes `client.deleteDatabase(dbId, opts?)` with the same `Idempotency-Key` discipline as `createDatabase`.
- **Core value:** Effortless UX, Bullet-proof
- **Why:** Free-tier abuse rules (`docs/phase-plan.md §8`) and the design-partner research in `docs/runbook.md §10` both push toward "cheap to undo / cheap to try" — but without a user-facing delete, every test session, abandoned exploration, and "wrong shape, redo" accumulates against the per-tenant DB-count cap. The sweep job (`apps/api/src/db-sweep/sweep.ts`) only evicts anon DBs by age + cap; authed tenants have no equivalent path, leaving operator cleanup as the only escape hatch. Two safety layers stop a stray click from wiping production data: (a) the typed-name confirmation in the UI (the user must read and reproduce the displayName, with a copy-paste affordance so a long disambiguated name doesn't motivate a typo workaround), and (b) the route's tenant scoping via `resolveDb`. Reusing `dropSchemaAndRegistry` rather than inlining DROP + DELETE upholds SK-HDC-011's "one primitive for compensation" rule; this PR makes the primitive **strict on error** so the user-delete path never falsely reports 204 when Neon refused the DROP (GDPR right-to-erasure compliance) — the create-rollback caller wraps the call in try/catch to preserve its prior best-effort semantics.
- **Consequence in code:**
  - `apps/api/src/db-create/neon-provision.ts` exports `stripDbPrefix(dbId)` (was module-private). The DELETE route uses it to compute the schema name; the export carries a comment explaining that the throw on a malformed id is exactly the boundary check SK-HDC-009 wants before constructing SQL identifiers downstream.
  - `apps/api/src/index.ts`: `DELETE /v1/databases/:id` added (placed after `POST /v1/databases`, before `POST /v1/chat/messages`). Uses the same dynamic-import pattern as the POST handler — `build-deps.ts` transitively pulls in `sql-validate-ddl.ts` whose top-level `await loadModule()` is the documented cold-start hang risk; static-importing here would force every `/v1/*` request to pay that cost. Delete imports **only `buildPgClient` + `resolveDatabaseUrl`** from `build-deps.ts`, not the full `buildDbCreateDeps` — the LLM router, embed deps, and recent-tables store are wasted construction on the delete path. The CORS `allowMethods` list adds `DELETE`. OTel span name: `nlqdb.databases.delete` with `nlqdb.databases.delete.outcome ∈ { ok | not_found | internal_error }`; `SpanStatusCode.ERROR` set on the error branch. On 500 the route returns a **typed `{ error: "internal_error" }` envelope** (matching `POST /v1/oauth/...` and `GET /v1/keys/:hash/status`) so SDK consumers get a structured `code` rather than the wire-layer `unknown_error` fallback. After `dropSchemaAndRegistry` the route runs `DELETE FROM api_keys WHERE db_id = ?` (in-process D1; SK-APIKEYS-001 binds `pk_live_*` rows to the dbId).
  - `apps/api/src/db-create/neon-provision.ts`: `dropSchemaAndRegistry` is now **strict** — it propagates errors instead of swallowing them, and the Postgres step uses `DROP SCHEMA IF EXISTS` so a partial-retry (schema already dropped, registry row still present) doesn't re-fail at that step. The create-rollback callsite wraps the call in `try/catch` to preserve its "original create error matters more than rollback hiccup" semantics; the new user-delete callsite relies on the strict propagation so a Neon-side DROP failure surfaces as a 500 (and the registry row stays so the user / operator can retry) rather than misleading the user with a 204 + silent orphan.
  - `apps/api/src/db-create/build-deps.ts`: exports `buildPgClient(connectionString)` and `resolveDatabaseUrl(env)` so the delete route can build a lean Postgres client without the full create-deps construction.
  - `packages/sdk/src/index.ts`: `NlqClient.deleteDatabase(dbId, opts?)` added next to `createDatabase`. URL-encodes the dbId (`encodeURIComponent`). Surfaces a typed `NlqdbApiError` with `code === "db_not_found"` on 404.
  - `apps/web/src/components/chat/LeftRail.tsx`: each rail row gains a `×` button (`aria-label="Delete <displayName>"`, `data-testid="delete-db-<slug>"`). A new `DeleteDbDialog` inside the same file owns the confirmation flow — it shows the `displayName` in a `<code>` block next to a "Copy" button, requires the typed input to match exactly before the "Delete" button enables, and dispatches `client.deleteDatabase` with a `web-delete-<dbId>-<ts>` idempotency key. **Deliberately no `placeholder` on the input** — rendering the exact text the user has to type would undercut the gate the whole feature exists to provide. **Focus management**: focus enters the input on mount, Tab cycles inside the dialog (focus trap so a stray Tab past Cancel can't reach the rail's × of another DB behind the scrim), Escape closes when not submitting, focus restores to the rail's × button on close (Cancel / Escape / success all funnel through one cleanup effect). **Error copy is delete-context-specific** via a local `messageForDelete` helper — `db_not_found` reads "This database is already gone — close to refresh." (catches the cross-tab race), `unauthorized` prompts re-auth, network errors say "Network error reaching nlqdb — try again." The error region carries `role="alert" aria-live="assertive"` so screen-reader users are notified. `ChatPanel.tsx` wires `onDeleted` so the active DB selection is cleared and per-DB history (`nlq_hist_<dbId>` in localStorage) is purged when the deleted row was the live one.
  - `apps/web/src/styles/global.css`: `.btn--danger` lives next to the canonical `.btn--accent` so future destructive UI (delete account, revoke API key) gets the modifier for free. `apps/web/src/styles/chat.css`'s delete-dialog backdrop uses `z-index: 60` so it wins the stack against the Cmd+K palette (`z-index: 50`) if both are open at once.
  - `tests/opencheck/tests.yaml`: the destructive cleanup arc is appended to the end-to-end flow so every passing run leaves the test tenant with zero hosted DBs. Two deletes — one before sign-out (active DB), one after sign-in-again (the remaining DB) — exercise both the dialog's typed-confirm path and the cross-session continuity.
  - **Surface parity (GLOBAL-003)**: HTTP + SDK + web app all updated in this PR. The CLI binary (`cli/`) is docs-only today — `docs/features/cli/FEATURE.md` tracks delete alongside create when the binary lands. The MCP server is intentionally read-only per **SK-MCP-002** (`nlqdb_query` / `nlqdb_list_databases` / `nlqdb_describe`; no `nlqdb_create_database` and by extension no `nlqdb_delete_database`); destructive ops live on user surfaces, not on agent surfaces. The `<nlq-data>` element (`packages/elements/`) is a query-consumption surface, not a provisioning surface — delete is not applicable there.
  - **Tests**: route-level integration tests in `apps/api/test/databases-delete.test.ts` cover the auth gate (`requireSession` rejects no-bearer + anon + leaked-`sk_live_*`) and the CORS preflight (advertises DELETE + `idempotency-key`). The `dropSchemaAndRegistry` strict-mode contract (Postgres-fail → propagates, D1-fail → propagates after DROP) is covered in `neon-provision.test.ts`. SDK-side 404 `db_not_found` typing is covered in `packages/sdk/test/client.test.ts`. The happy-path 204 wire shape is not unit-tested at the route level because that path requires a real Better Auth session — the integration tests rely on session-less 401 + CORS coverage, and the underlying primitives are fully stubbed in unit tests.
  - **Known limitations (not blocking for this PR)**:
    - **No per-endpoint rate limit on `DELETE /v1/databases/:id` or its sibling `POST /v1/databases`.** Both inherit only `requireSession` (no anon traffic) + tenant scope (a user can only thrash their own DBs). `SK-HDC-008` covers the per-account D1 limiter on the `/v1/ask` create path; the dedicated CRUD routes don't yet pay through that gate. Worth a follow-up `SK-RL-NNN` in `docs/features/rate-limit/FEATURE.md` once a user actually thrashes; not on the Phase 1 critical path.
    - **No concurrency coordination between an in-flight `/v1/ask` and a concurrent DELETE on the same dbId.** A query racing with the drop lands an opaque Neon error (`relation does not exist`). Acceptable for Phase 1 (free tier, no SLA); worth carving a structured "DB deleted while query in flight" envelope if user reports surface.
    - **No `statement_timeout` on the `DROP SCHEMA CASCADE`.** Phase-1 DBs are bounded by SK-HDC-005's per-tenant cap so the CASCADE's table count is small in practice; if Phase-N hosted DBs grow large, a server-side `SET LOCAL statement_timeout` would prevent a runaway drop from blocking the Neon HTTP round trip.
    - **`DELETE FROM api_keys` runs after `dropSchemaAndRegistry` and is not transactional with it** (D1 and Neon are separate datastores; no two-phase commit). If the api_keys DELETE fails after the registry row is gone, the route returns 500 and the user has orphan `pk_live_*` keys. Those keys are useless without the DB they reference (the route auth on them resolves the dbId via D1 and 404s), but they're cosmetic clutter until a sweep cleans them up. Acceptable for Phase 1.
- **Alternatives rejected:**
  - **No server-side route; rely on the sweep to evict by inactivity.** Sweep only covers anon tenants today (`db-sweep/sweep.ts` WHERE clause pins `tenant_id LIKE 'anon:%'`). Adding authed-tenant inactivity sweep is a strictly larger change and forces users to wait for retention to expire instead of releasing a slot immediately when they're done.
  - **Soft delete (`deleted_at` column, hide from list).** Doubles the storage cost — both the D1 row and the Postgres schema persist — and contradicts the user's intent ("I'm done with this DB"). Recovery from accidental delete is the UI's typed-name confirmation's job, not a server-side undelete window. A future Phase-N "trash" surface can be carved if a real user need surfaces; today's anon-mode 90-day TTL is already a 90-day window-of-mercy for the anon tier, and the authed tier has no equivalent design partner ask.
  - **Inline `DROP SCHEMA` + `DELETE` in the DELETE handler.** Forks the rollback primitive — SK-HDC-011 explicitly forbids this for exactly this reason. One primitive, one bug surface.
  - **Confirm via a server-side `confirm: "<displayName>"` body param instead of UI-only.** Pushes UX policy into the API. The wire call is a stateless intent-capture; the typed-name gate is a UX concern that the surface owns. A power user calling the API directly via `nlq` / curl has already opted into "I know what I'm doing"; the typed-name speed-bump is for the chat surface, not the protocol.
  - **`POST /v1/databases/:id/delete` with `{ confirm: "<name>" }`.** Non-RESTful and conflates intent capture with the action. The user-question that prompted this design picked `DELETE /v1/databases/:id` for conventional reasons; the confirmation gate sits in the UI where it belongs.
