# SK-HDC-019 ‚Äî Pre-validate sample rows and drop only the uninsertable ones, salvaging the rest

- **Decision:** Before the first `provisionDb` attempt, `orchestrateDbCreate` runs `pruneUninsertableSampleRows(plan)` (pure, in `apps/api/src/db-create/sample-rows.ts`) and provisions with `plan.sample_rows` reduced to only the rows that can actually insert. A row is dropped **only when it provably cannot insert** against the plan's own declared constraints, walking rows in plan order so a dropped parent cascades to its dangling children: unknown table / unknown column (class 42), a NOT NULL column (explicit `nullable:false` or a primary-key member) with no `DEFAULT` and an absent/null value (23502), an uncoercible value for `integer`/`bigint`/`numeric`/`real`/`double_precision`/`uuid`/`boolean` (22P02), or a non-null foreign-key value with no matching parent row accepted earlier in plan order (23503). Coercible string forms (`'2'` into integer, `'1.5e3'` into numeric, `'NaN'`, the Postgres boolean literals) are kept, and FK matching uses string-coerced equality, so a row Postgres would accept is **never** dropped. A clean plan prunes nothing ‚Äî the happy path is byte-for-byte unchanged. [`SK-HDC-018`](./SK-HDC-018-sample-insert-graceful-degradation.md) stays as the residual floor: any failure this static pass cannot model (a `CHECK`, a unique collision, an unmodeled type form) still degrades to the un-seeded retry, never a 500.
- **Core value:** Bullet-proof, Goal-first, Engine quality
- **Why:** SK-HDC-018 made a bad seed row non-fatal, but its floor is **all-or-nothing**: a single constraint-violating row aborts the one atomic insert ([`SK-HDC-012`](../FEATURE.md#sk-hdc-012)) and the retry ships `sample_rows: []`, so the stranger's first value collapses from a seeded demo to an *empty* DB. That collapse is exactly what the `seeded_ok_ratio` seed-quality probe measured the cost of (0.25 on the first 2026-06-10 run; ~0.6‚Äì0.8 after the SK-LLM-033 prompt, with a persistent degrading tail ‚Äî "a meal planner for couples" empties on every run, its `meals‚Üícouples` / `ingredients‚Üímeals` FKs the canonical forward-reference miss). The prompt ([`SK-LLM-033`](../../llm-router/decisions/SK-LLM-033-schema-infer-insertable-sample-rows.md)) is probabilistic and can't be unit-tested; a deterministic pre-validation is the cheap, sound complement that keeps the *valid* rows whenever the model emits a mixed batch ‚Äî on a 13-row seed with one bad row the stranger now sees 12 rows instead of 0. Pure scaffolding that compounds with the model ([`GLOBAL-026`](../../../decisions/GLOBAL-026-llm-strategy-byollm-hosted-premium.md)).
- **Consequence in code:** New pure module `apps/api/src/db-create/sample-rows.ts` exports `pruneUninsertableSampleRows(plan): { rows, dropped }` (no IO, no input mutation). `orchestrate.ts` calls it once where `provisionPlan` is initialized, swaps in the pruned `sample_rows` only when something was dropped, and logs one structured `console.warn({msg:"provision_sample_rows_pruned",dropped,kept,reasons})` line on that branch. `schemaHash` / `schemaText` still derive from the original `plan` (sample rows are not part of the schema fingerprint), so plan-cache identity is unchanged; the response's `sampleRows` reports the actually-seeded set, so it never claims rows the DB doesn't hold. `sample-rows.test.ts` is the measured before/after: the salvage case (one NOT-NULL-violating row of four ‚Üí 3 kept, was 0 under the all-or-nothing retry), the clean-plan no-op, the cascade, FK string/number coercion (no false drop), and each drop reason. Deterministic and free ‚Äî no extra DB round-trip, no LLM call. The remaining un-modeled failures keep flowing through SK-HDC-018.
- **Alternatives rejected:**
  - **Keep SK-HDC-018's all-or-nothing floor alone.** One bad row of N still costs the stranger the other N‚àí1 seeded rows ‚Äî the empty-DB tail `seeded_ok_ratio` keeps measuring. The floor is necessary (residual unmodeled failures) but not sufficient.
  - **Salvage on the retry instead of pre-emptively.** Would need an extra provision round-trip (fail ‚Üí prune ‚Üí re-provision) and risk a third attempt when the static prune can't find the offending row. Pre-validating up front is one in-memory pass, no extra DB call, and leaves SK-HDC-018's "retry once" invariant untouched.
  - **Drop the offending *column* rather than the row.** Re-introduces the NOT-NULL question (a stripped column might itself be required) and silently changes the row's shape. Dropping the whole provably-bad row is simpler (P5) and the cascade keeps the data model coherent.
  - **LLM auto-repair the row.** Rejected for the same reasons as SK-HDC-017/018 ‚Äî a second LLM call adds latency, cost, and a new failure surface for a decorative row. Static pruning is deterministic and free.
