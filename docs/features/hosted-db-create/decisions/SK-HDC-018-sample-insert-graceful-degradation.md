# SK-HDC-018 — A constraint-violating sample row degrades to an un-seeded DB, never a 500

- **Decision:** When `provisionDb` fails with `reason: "sample_insert_failed"` (SQLSTATE class 22/23 — an LLM-authored seed row that violates its own schema's FK / NOT NULL / type, per [`SK-HDC-017`](./SK-HDC-017-provision-sqlstate-fidelity.md)), `orchestrateDbCreate` retries the provision **once** with `plan.sample_rows = []`. The retry reuses the same `dbId` / `schemaName` (the first attempt rolled back, so the ids are free and the already-compiled DDL still matches the schema name) and the same compiled DDL — only the seed-row INSERTs are dropped. The response's `sampleRows` reports the *actually-inserted* set, so an un-seeded DB returns `sampleRows: []` and never claims rows it doesn't hold. A second `sample_insert_failed` (or any other non-`schema_already_exists` reason) surfaces `provision_failed` as before — the retry is bounded to one extra attempt, no LLM repair, no loop.
- **Core value:** Bullet-proof, Goal-first, Honest latency
- **Why:** A stranger's first-value path (describe a goal → `/v1/ask` → auto-provision) means their first query *is* their first impression. The free chain ([`GLOBAL-026`](../../../decisions/GLOBAL-026-llm-strategy-byollm-hosted-premium.md)) authors `sample_rows` at default temperature with no insertability contract, so it intermittently emits a row that breaks the very FK / NOT NULL constraint it just declared; because schema + RLS + seed rows share one atomic transaction ([`SK-HDC-012`](../FEATURE.md#sk-hdc-012)), that single decorative row rolled the **entire** create back to `HTTP 500` and the stranger got *no database at all*. Observed on the deployed surface 2026-06-08: 5/5 probes returned `{kind:"provision_failed",reason:"sample_insert_failed"}` for "a meal planner for couples" — SK-HDC-017's class mapping correctly attributed it to the seed-insert phase, not infra. The schema the user asked for is sound; only the demo data failed. Dropping the seed rows hands them a working, schema-complete database to fill, which is strictly better than a 500. This mirrors the spirit of the [`GLOBAL-033`](../../../decisions/GLOBAL-033-resolve-open-questions-via-ladder.md)-resolved multi-statement-boundary stance recorded in this feature's Open questions ("a Workers-AI rate-limit can't cost the user their DB; an un-embedded DB degrades semantic search, not data loss") — an un-seeded DB degrades the first-value *demo*, not the database itself.
- **Consequence in code:** `apps/api/src/db-create/orchestrate.ts` step 5 tracks `provisionPlan` (the plan actually handed to `provision`) and a one-shot `sampleRowsDropped` flag; on `sample_insert_failed` it sets `provisionPlan = { ...plan, sample_rows: [] }`, logs one structured `console.warn({msg:"provision_sample_rows_dropped",dbId,dropped})` line (non-fatal, one line, only on the rare fall-through — no per-row spam), and re-enters the loop. The `schemaHash` / `schemaText` written to the registry still derive from the original `plan` (sample rows are not part of the schema fingerprint), so plan-cache identity is unchanged. The atomic-transaction invariant is preserved — *each* attempt is still one `pg.transaction([...])` (SK-HDC-012), so there is never a half-provisioned schema. The response returns `sampleRows: provisionPlan.sample_rows`. Unit tests in `orchestrate.test.ts` pin: (a) `sample_insert_failed` then success → `ok`, 2 provision calls, the 2nd called with an empty `sample_rows`, response `sampleRows: []`; (b) persistent `sample_insert_failed` → `provision_failed` after exactly 2 calls (no infinite retry). The stranger then gets a reachable, queryable DB with `row_count: 0` — degraded-but-present, surfacing the seed-quality gap without re-introducing the 500. The complementary engine-quality fix — instructing the inferrer to emit insertable seed rows — ships as `SK-LLM-033` in `llm-router`.
- **Alternatives rejected:**
  - **Keep the atomic 500 (status quo).** Costs the stranger their database over a decorative row the product never needed — the worst possible first impression on the funnel that reaches first-value.
  - **Make seed-row inserts non-atomic / best-effort within the original transaction.** Contradicts the [`GLOBAL-033`](../../../decisions/GLOBAL-033-resolve-open-questions-via-ladder.md)-resolved boundary ("schema + sample rows + `databases` row commit atomically") recorded in this feature — that would be a silent decision reversal (CLAUDE.md P1). The retry keeps every attempt atomic instead.
  - **LLM auto-repair the offending row and retry.** Exactly what [`SK-HDC-017`](./SK-HDC-017-provision-sqlstate-fidelity.md) rejected as out-of-scope / against P5 — a second LLM call adds latency, cost, and a new failure surface for a row the user never asked for. Dropping it is deterministic and free.
  - **Topologically sort + FK-prune the seed rows in the compiler.** This decision keeps the all-or-nothing floor; the sound, all-class version of seed pruning ships separately as [`SK-HDC-019`](./SK-HDC-019-deterministic-sample-row-salvage.md) — it runs on the cold create path (not the `/v1/ask` hot path) and *does* cover dangling FK values, NOT-NULL gaps, and bad types (the gaps that made a compiler-side sort insufficient on its own), so the two compose: SK-HDC-019 salvages the valid rows up front, this floor catches whatever it can't statically model. The prompt contract (SK-LLM-033) remains the cheapest place to raise seed validity.
