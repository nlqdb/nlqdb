# SK-APIKEYS-011 — `DELETE /v1/keys/:id` is hard-revoke (sets `revoked_at = now`); rotation grace lives in SK-APIKEYS-005

- **Decision:** `DELETE /v1/keys/:id` is the revoke verb. It is session-gated, tenant-scoped, and idempotent per RFC 9110 — a re-DELETE on an already-revoked key returns `200 { ok: true, alreadyRevoked: true }` rather than 404, so a retried call doesn't have to special-case the race ("did someone else revoke it first?"). A key id that doesn't exist or belongs to another tenant returns `404 key_not_found` — identical envelopes for both so cross-tenant existence doesn't leak. The mutation sets `revoked_at = unixepoch()` on the row; the row stays so `getKeyStatusByHash` (SK-MCP-014) can answer "revoked" vs "unknown". Propagation to live MCP sessions is ≤ 1 s via the DO revalidation cache.
- **Core value:** Bullet-proof, Effortless UX, Simple
- **Why:** RFC 9110 says DELETE is idempotent; Stripe and every modern API key SDK matches that posture (no `Idempotency-Key` needed; the verb itself is the dedupe contract). The 60-day rotation grace + webhook plus the events-pipeline emission belong to [`SK-APIKEYS-005`](SK-APIKEYS-005-rotation-grace.md) and require an events-pipeline rotation event the slice doesn't have yet. Shipping hard-revoke first means: (a) leaked keys can be killed today; (b) `SK-MCP-009`'s "revocation ≤ 1 s" contract has a surface that exercises it (without revoke, the DO revalidation probe never fires the revoked branch); (c) the dashboard's revoke button can ship before rotation lands.
- **Consequence in code:** `apps/api/src/api-keys.ts::revokeKeyById` is the only mutation path; route is `DELETE /v1/keys/:id` in `apps/api/src/index.ts`. SDK exposes `client.revokeKey(id)` returning `{ ok, alreadyRevoked }`; CLI exposes `nlq keys revoke <id>`. The 404-vs-403 ambiguity is deliberate ([`SK-HDC-016`](../../hosted-db-create/decisions/SK-HDC-016-delete-database.md) uses the same posture for `DELETE /v1/databases/:id`). The DO revalidation cache (SK-MCP-014) already filters `revoked_at IS NULL` at the source, so no extra cache-bust is needed here.
- **Alternatives rejected:**
  - `POST /v1/keys/:id/revoke` — works, but isn't idempotent by HTTP semantics; would need `Idempotency-Key` plumbing to match the DELETE behavior we get for free.
  - 404 on already-revoked — forces callers to grep error strings to distinguish "race lost" from "wrong id," for no security benefit.
  - Hard-delete the row (no `revoked_at`) — `getKeyStatusByHash` (SK-MCP-014) can no longer distinguish revoked from unknown; the DO would surface the same "auth failed" envelope for both, breaking the SK-MCP-006 recoverable error contract.
  - Ship 60-day rotation grace in the same slice — couples revoke to a webhook/events-pipeline path that isn't ready; blocks the dashboard revoke button on the rotation feature.
