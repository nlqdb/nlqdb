# SK-APIKEYS-012 — Dashboard key-management UI at `/app/keys` with copy-once mint + confirm-revoke

- **Decision:** The dashboard key-management UI lives at `/app/keys` (under the authed `/app/*` namespace). It renders three affordances on top of the existing endpoints: a key list grouped into Active / Revoked sections (`GET /v1/keys`), a "New key" modal that mints `sk_live_*` and surfaces the plaintext exactly once inside a copy-once disclosure (`POST /v1/keys` per [`SK-APIKEYS-007`](SK-APIKEYS-007-mint-via-post-keys.md) + [`SK-APIKEYS-002`](SK-APIKEYS-002-argon2id-no-plaintext.md)), and a Revoke action gated by a single-confirm modal (`DELETE /v1/keys/:id` per [`SK-APIKEYS-011`](SK-APIKEYS-011-hard-revoke.md)). Minting `sk_mcp_*` is intentionally **not** exposed in this UI — those keys flow through the OAuth-callback server-side mint ([`SK-APIKEYS-009`](SK-APIKEYS-009-sk-mcp-server-side-mint.md)) or `nlq mcp install`'s deep-link (`SK-MCP-007`). Listing + revoking `sk_mcp_*` rows is supported. The SDK gains `mintKey(req)` to round out the trio with `listKeys()` / `revokeKey()`.
- **Core value:** Effortless UX, Bullet-proof, Honest latency
- **Why:** The dashboard is the user's only path to mint `sk_live_*` keys for the CLI (`NLQDB_API_KEY`) and server-to-server callers — without it the open question in this feature's index file becomes a permanent gap. **Display-once** is the dominant 2026 industry pattern (Stripe, GitHub, Anthropic, AWS, Linear) because rendering the plaintext perpetually adds passive leak surface (screenshots, screen-share, shoulder-surfing) with zero offsetting value over "mint a fresh one + revoke the old" — and the plaintext was never persisted server-side anyway ([`SK-APIKEYS-002`](SK-APIKEYS-002-argon2id-no-plaintext.md)). **Single-confirm revoke** (no typed-name gate like [`SK-HDC-016`](../../hosted-db-create/decisions/SK-HDC-016-delete-database.md)) is right-sized: a finger-slip revoke is recoverable in one click by minting a fresh key, where a finger-slip db.delete drops every row in the schema permanently. The two flows share the same modal chrome (`.keys-dialog`) so the surface feels coherent without inheriting the typed-name cost on the recoverable case.
- **Consequence in code:**
  - Page: `apps/web/src/pages/app/keys.astro` (inline auth-guard mirroring [`SK-AUTH-016`](../../auth/decisions/SK-AUTH-016-merged-web-api-worker.md) / [`SK-WEB-009`](../../web-app/FEATURE.md)).
  - React island: `apps/web/src/components/keys/KeysPanel.tsx` (orchestrates list + mint + revoke dialogs). Pure helpers in `apps/web/src/components/keys/group.ts` are unit-tested in `apps/web/src/components/keys/group.test.ts`.
  - Dialog primitives: `apps/web/src/lib/dialog.ts` exports `useFocusTrap` + `useRestoreFocusOnUnmount`, shared with `DeleteDbDialog` (`SK-HDC-016`). Mint dialog uses a `useRef` in-flight guard so a rapid second Enter-in-input can't race past the `submitting` state and fire two `POST /v1/keys`; both dialogs abort the in-flight request on unmount via `AbortController`.
  - Styles: `apps/web/src/styles/keys.css` reuses the neo-brutalist + terminal tokens from `global.css`; the `.keys-dialog` chrome mirrors `.delete-db-dialog` so destructive surfaces feel sibling-like. The screen-reader-only `.visually-hidden` utility lives in `global.css` so any surface can consume it.
  - SDK: `packages/sdk/src/index.ts` adds `MintKeyRequest` / `MintKeyResult` types and `client.mintKey(req)` (`POST /v1/keys`); plaintext lands on the response exactly once and is not cached anywhere downstream. Test coverage in `packages/sdk/test/client.test.ts`.
  - Wiring: `apps/web/src/components/chat/ChatPanel.tsx` retargets the Cmd+K "API keys" command from the dangling `/app/settings` stub to `/app/keys`; `packages/mcp/src/tools.ts`'s `auth_required` envelope now points at `https://app.nlqdb.com/app/keys` so MCP hosts surface a working link.
  - Clipboard: writes via `navigator.clipboard.writeText` (HTTPS-only — the merged Worker is always HTTPS per [`SK-AUTH-016`](../../auth/decisions/SK-AUTH-016-merged-web-api-worker.md), so no `document.execCommand` fallback). The plaintext stays selectable via `user-select: all` in the code block, so a clipboard rejection still lets the user keyboard-copy.
  - Accessibility: WAI-ARIA APG disclosure pattern — `role="dialog"` + `aria-modal="true"`, focus-trap mirrors `DeleteDbDialog`, screen-reader-only `.visually-hidden` prefix on the `…last4` code block reads "Key ending in …abcd" without breaking the visual brevity.
- **Alternatives rejected:**
  - **Retrievable keys (Resend/Supabase pattern).** Would require storing plaintext (encrypted-at-rest, decryptable on-demand) which directly contradicts [`SK-APIKEYS-002`](SK-APIKEYS-002-argon2id-no-plaintext.md)'s "no plaintext retrieval" promise and adds a leak surface for zero functional benefit on a recoverable-by-rotate posture.
  - **Inline plaintext reveal on the list page.** Same screenshot / screen-share leak surface as retrievable keys, plus it would force re-implementing "show value" gating per key. The display-once modal hands the plaintext to the user once at mint and then drops it from component state — no list row ever carries the full value.
  - **Typed-name confirm on revoke.** Patronising for a recoverable op (mint a fresh one to recover). The cost of an accidental revoke is one extra round-trip; the cost of a friction-laden revoke flow is users who leak keys because revocation feels like a chore.
  - **Top-level `/keys` URL** (matching `packages/mcp/src/tools.ts`'s pre-existing string verbatim). Mixes authed and anon top-level routes; the MCP message is updated to point at `/app/keys` so the convention "authed pages live under `/app/*`" holds.
  - **`/app/settings/keys` (settings-hub layout).** Adds a hub page for one item today, and the future "billing" / "team" surfaces aren't scoped yet. Promote to a hub only when the second top-level surface lands.
  - **SDK-only (no UI).** The CLI already wraps `listKeys` / `revokeKey` ([`SK-APIKEYS-010`](SK-APIKEYS-010-list-endpoint.md) / [`SK-APIKEYS-011`](SK-APIKEYS-011-hard-revoke.md)) but power-users with browser-only workflows (and the `nlq mcp install` deep-link target per `SK-MCP-007`) still need the dashboard. Shipping just the SDK widens the gap rather than closing it.
