# SK-APIKEYS-010 — `GET /v1/keys` lists the caller's inventory in one envelope; no pagination in v1

- **Decision:** `GET /v1/keys` returns the caller's full key inventory as `{ keys: KeyRecord[] }`. The list is session-gated (a leaked `sk_live_` cannot enumerate sibling keys — same posture as `POST /v1/keys`) and tenant-scoped at the SQL layer (`WHERE tenant_id = ?`). Active rows sort before revoked, then by `created_at DESC`, so a single slice covers both the "active keys" and "revoked keys" sections the dashboard renders. Plaintext is never present — `last4` is the only display field ([`SK-APIKEYS-002`](SK-APIKEYS-002-argon2id-no-plaintext.md)). No `?status=` filter, no cursor pagination.
- **Core value:** Simple, Bullet-proof, Effortless UX
- **Why:** Per-tenant key counts are bounded (<50 typical — three MCP hosts × three devices, one or two `sk_live_` for CI, a handful of `pk_live_` per DB). Cursor pagination at this scale adds round-trips and a cache-coherency problem ("did the row I just revoked move pages?") without solving anything. One envelope keeps the dashboard fast (single fetch hydrates both groups) and keeps the CLI verb a one-liner. The `(revoked_at IS NOT NULL), created_at DESC` ordering is stable so re-fetching after a revoke doesn't reshuffle the active section.
- **Consequence in code:** `apps/api/src/api-keys.ts::listKeysByTenant` is the only query path; route is `GET /v1/keys` in `apps/api/src/index.ts`. SDK exposes `client.listKeys()`; CLI exposes `nlq keys list`. `KeyRecord` is shared across SDK + CLI wire shapes. **Upgrade trigger:** if any single tenant's `api_keys` row count crosses 200 (revoked-included), evaluate `?status=active` filter + cursor pagination; the index `api_keys_tenant` already covers the new sort path.
- **Alternatives rejected:**
  - Cursor pagination day 1 — over-engineered for the working-set size; adds cache-coherency complexity that hurts the dashboard.
  - Two endpoints (`/v1/keys/active` + `/v1/keys/revoked`) — doubles round-trips for the dashboard; gives the operator no view of the join between them.
  - Return only active rows, expose revoked rows on `/v1/keys/:id` — the dashboard's "revoked keys" section becomes N+1 fetches; surface tax with no security upside.
