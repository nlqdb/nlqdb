- **Core value:** Effortless UX, Goal-first, Seamless auth

- **Why:** The hero ends with *"you have a database."* The next question every agent builder asks â€” within seconds â€” is *"how does my agent talk to it?"* Today the answer is three friction-laden steps: read the docs, find the connector URL, paste it into the host's config file (in the right shape, in the right location, against the right wrapper key). Most agent IDEs now ship a deep-link install scheme exactly to collapse that to one click; for the hosts that don't, copying a paste-ready JSON block (with the correct wrapper key and field name for *that specific host* â€” `mcpServers` vs `context_servers`, `url` vs `serverUrl`) is the next-best step. A single `<McpInstall>` component encapsulates the per-host shape knowledge so a docs-stale rendering of the install path can never ship from a one-off page edit. Three venues are load-bearing: `/agents` is the page whose whole pitch is *agent memory over MCP* â€” install adjacency is on-message, not interruption (the existing connect card already implies this; `SK-WEB-016` formalises it with the deep-link buttons replacing the manual copy-config UI). The post-create surface catches the user at peak engagement â€” they just saw their schema render â€” and converts "I have a DB" into "my agent can talk to it" without leaving the page. `/integrations` is the canonical reference page; replacing the static four-cell host grid with the installer turns documentation into action. `SK-WEB-002` (one input above the fold) is preserved â€” the install row sits *after* the CTA on `/agents`, never beside it, and never appears on the marketing homepage. `SK-WEB-015` accent budget is preserved â€” one promoted lime CTA per row, the rest `.cta--ghost`. `SK-ANON-012`'s no-live-anon-key contract is preserved â€” the placeholder key + sign-in CTA is the honest path (inlining a real anon key would 401 because the 1-call cap was already burned by the create call). `SK-WEB-007`'s rule that only the chat inlines a working key is preserved â€” the McpInstall configs ship the placeholder until a real `pk_live_` arrives from the per-DB key path.

- **Consequence in code:** `apps/web/src/lib/mcp-install.ts` is the single source of truth for the per-host install knowledge â€” `buildMcpHosts(mcpUrl)` returns the four entries (`id`, `status`, `href` or `config`, `pasteHint`, `versionHint`), and exports `MCP_ENDPOINT_URL` / `MCP_SERVER_ROUTE` / `PLACEHOLDER_KEY` / `PROMOTED_HOST`. Both venues import it so they can't drift: `McpInstall.astro` (SSR, vanilla-JS clipboard handler, zero framework weight) and the `McpInstallView` React mirror inside `CreateForm.tsx` (the Astro component scopes its styles, so the React island can't borrow them and renders an equivalent surface inline). The Cursor deep-link is `base64(JSON.stringify({url: MCP_ENDPOINT_URL}))` URL-encoded into the `config` query param; `toBase64` feature-detects `Buffer` (SSR) vs `btoa` (browser) so both venues produce identical URIs. `mcp-install.test.ts` decodes every host's connection URL and asserts its path equals `MCP_SERVER_ROUTE` â€” catching root, a doubled `/mcp/mcp`, or any other drift. Clipboard failure is swallowed silently (the `/integrations` configs stay visible). The demand signal reuses the `home.snippet_copied` LogSnag envelope with `surface = mcp_install_<host>` so the install funnel reads on the embed-snippet dashboard (`SK-WEB-010` / `GLOBAL-024`). Venues: `/agents` hero (compact, under the form, `--rule` hairline above), post-create `CreateResultView` (after the schema preview, before `<CreateSnippetView>`), `/integrations` MCP-hosts section (`compact={false}`, the old four-cell grid dropped). PRs that add a fifth host without re-running the Phase 1 research (deep-link scheme + min version + fallback config shape) are rejected â€” every host entry must cite its source. PRs that move lime onto a second button per row are rejected â€” one promoted CTA per row is the `SK-WEB-015` accent ladder.

- **Alternatives rejected:**
  - **Render buttons next to the homepage hero input.** Rejected because `SK-WEB-002` keeps the hero to one input above the fold; the install row would compete with the "what are you building?" affordance and dilute the goal-first inversion. The post-create surface (which IS after the CTA) is the right venue for the marketing flow.
  - **Multi-step text instructions, one set per host.** Rejected because the per-host config shape is non-trivially divergent (Claude's `mcpServers` + `url`, Windsurf's `mcpServers` + `serverUrl` + `/mcp` suffix, Zed's `context_servers` + `url`, plus three different paste locations), and host versions ship breaking changes to those shapes regularly â€” a copy-paste-correct-shape button captures the divergence in one place where it can be updated once.
  - **Inline a real anon `pk_live_â€¦` in the McpInstall configs.** Rejected because `SK-ANON-012`'s per-device cap is consumed by the create call; the anon key would 401 on the first tool call from the host. Silent-broken-install is a worse UX than the placeholder + honest sign-in nudge.
  - **One button per host with the same label "Install" and a host icon.** Rejected because the label "Add to Claude" / "Add to Cursor" carries the host name as scannable text â€” icons read worse on screen readers and lose the brand recognition cue that drives the click. The version hint (e.g. "Cursor current stable, 2026+") rides the `title` attribute so the affordance stays honest without inflating the row's visual weight.
  - **Publish to the Windsurf MCP Registry and use `windsurf://windsurf-mcp-registry?serverName=nlqdb` as the deep-link path.** Rejected for the foundation â€” the registry submission is a separate workstream (review process, ownership, takedown risk). Once published, swapping Windsurf from `status: "fallback-only"` to `status: "deep-link"` is a one-line change in `HOSTS`.
