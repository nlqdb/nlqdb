# SK-WEB-021 — `/architecture`: interactive 3D system map on its own route, never on `/`

- **Decision:** The system architecture gets one marketing route, `/architecture/`: a three.js interactive 3D map (React island `apps/web/src/components/ArchitectureMap.tsx`) above a server-rendered prose walkthrough, both rendered from one data module (`apps/web/src/data/architecture.ts`, which mirrors `docs/architecture.md` §2 — architecture.md wins on disagreement). **Zoom level IS the level of detail:** far shows three group cards (*You ask → the engine → your data*) with aggregate flow edges; dollying in crossfades to the full node graph (surfaces, router, auth, plan cache, compiler, validator, executor, engines) with labelled hit/miss edges; clicking a piece opens an info card and accent-highlights its edges while dimming the rest. three.js is **dynamic-imported on this route only**, after hydration — no other page fetches it. Scroll-trap guards: wheel-zoom activates only after a first click, and one-finger touch keeps scrolling the page (two fingers drive the map). Idle sway + edge pulses are `prefers-reduced-motion`-gated. The map never ships on `/` — not as a background, not as a hero.
- **Core value:** Simple, Effortless UX, Bullet-proof
- **Why:** "How does it work?" is a trust question for a pre-beta whose pitch is *a backend that doesn't exist* — showing one engine, with a validator standing between the model and your data, as an explorable map answers it faster than prose, and the zoom-as-disclosure lets a 14-node graph stay unoverwhelming (three cards first, depth on demand). On `/` the same scene would contradict three standing decisions at once: SK-WEB-018's two-door IA, SK-WEB-020's one-motion-moment budget, and SK-WEB-001's static-first / Lighthouse-100 posture (~170 KB gz of WebGL on the highest-traffic route).
- **Consequence in code:** `pages/architecture.astro` (page + overlay styles) + `components/ArchitectureMap.tsx` (ErrorBoundary-wrapped island) + `data/architecture.ts` (graph, tested by `architecture.test.ts` — dangling edges fail CI). The prose sections render from the same data module, so no-JS visitors, crawlers, and WebGL-less browsers lose nothing; the island degrades to a one-line pointer at the prose. Route is listed in the footer, `sitemap.xml.ts`, and `llms.txt.ts`. PRs that import three.js on any other route, hydrate it eagerly, or re-add a full-circle auto-orbit are rejected.
- **Alternatives rejected:**
  - **Homepage 3D background that turns interactive on click** — explicitly considered; violates the SK-WEB-018 / SK-WEB-020 / SK-WEB-001 trio and spends the site's whole JS budget on its most latency-sensitive route.
  - **react-three-fiber** — a second renderer + reconciler dependency for one island; vanilla three keeps it one library and the island disposable.
  - **Static SVG/2D diagram** — no depth-on-demand; overview-to-detail progressive disclosure is the point.
  - **Docs-site page instead** — the audience is pre-signup trust-building; docs.nlqdb.com is post-adoption.
