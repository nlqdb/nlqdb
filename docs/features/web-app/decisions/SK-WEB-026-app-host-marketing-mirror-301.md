# SK-WEB-026 ‚Äî Merged app host 301-redirects the marketing content trees to the canonical host

- **Decision:** The merged app host `app.nlqdb.com` (`nlqdb-api`, `SK-AUTH-016`) serves the **same** `apps/web/dist` build as the canonical marketing host `nlqdb.com`, so its marketing content trees (`/blog`, `/solve`, `/vs` ‚Äî ~105 of the ~115 indexable surfaces) are crawlable duplicates. A thin front-controller in `apps/api/src/index.ts` (`marketing-mirror.ts`) **301-redirects those trees on `app.nlqdb.com` to `https://nlqdb.com/<same-path>`**, consolidating SEO authority on the one canonical host. Only the wildcard-covered content trees are matched; the product (`/app/*`), auth (`/auth/*`), API (`/v1/*`, `/api/auth/*`), the root `/`, and static marketing singles (`/pricing`, `/architecture`, ‚Ä¶) are untouched. `apps/api/wrangler.toml` scopes `run_worker_first` to exactly those tree prefixes so every other path keeps its zero-cost asset-first / Hono route. On preview / `*.workers.dev` hosts the front-controller serves the built asset (no duplication to fix there, and Hono has no marketing routes).
- **Core value:** Free, Simple, Bullet-proof
- **Why:** GSC (2026-07) surfaced `app.nlqdb.com/blog/postgres-validator-rejects-valid-clickhouse-sql/` indexed at page-1 (position 8) ‚Äî a duplicate of the canonical `nlqdb.com/blog/‚Ä¶`. The pages already carry a correct `rel=canonical` to `nlqdb.com` and the sitemap lists only `nlqdb.com` URLs, yet Google indexed the app-host copy anyway (`rel=canonical` is a hint, not a directive). As indexation grows across the 115 submitted URLs, a full second crawlable host splits crawl budget and ranking authority ‚Äî directly suppressing the canonical host's impression breadth, which is the acquisition bottleneck (scorecard row #7). A 301 is the one signal Google treats as authoritative for host consolidation. The app host has nothing of its own to rank (product is behind auth; API returns JSON), so redirecting its marketing trees has no downside.
- **Consequence in code:** `apps/api/src/marketing-mirror.ts` exports the pure `marketingMirrorRedirect(url)` + `isMarketingMirrorPath(pathname)` (unit-tested in `marketing-mirror.test.ts`); the worker `export default.fetch` calls them before `app.fetch`. `apps/api/wrangler.toml` `[assets]` carries `run_worker_first = ["/blog", "/blog/*", "/solve", "/solve/*", "/vs", "/vs/*"]`. Adding a NEW top-level marketing content tree means adding its prefix to both the `run_worker_first` array and `MARKETING_MIRROR_PREFIXES` (they must stay in sync). Consistent with `SK-WEB-023`'s index-open posture ‚Äî it *enforces* the self-referential canonical by removing the non-canonical host, rather than adding a `noindex`.
- **Alternatives rejected:**
  - `X-Robots-Tag: noindex` on the app host's marketing pages ‚Äî needs the same `run_worker_first` intercept but leaves the duplicate URLs live (weaker than a 301, which forwards any inbound link equity to the canonical).
  - `robots.txt` `Disallow` on the app host ‚Äî the file is shared static (`../web/dist`), so it can't be host-differentiated without the same worker intercept; and `Disallow` blocks crawl so Google can't even see the canonical, leaving stale index entries. Google's own guidance prefers 301 / `noindex` over `Disallow` for de-duplication.
  - A Cloudflare dashboard Redirect Rule ‚Äî a console click; not agent-movable (`GLOBAL-033`).
  - `run_worker_first = true` (worker in front of every request) ‚Äî routes all product-asset traffic through the Hono app; larger blast radius for a bounded SEO fix.
