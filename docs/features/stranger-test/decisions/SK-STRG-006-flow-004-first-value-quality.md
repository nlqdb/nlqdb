# SK-STRG-006 — FLOW-004 walker asserts first-value *quality* on the HTTP 200, not just reachability

- **Decision:** When the invite probe in `scripts/flow-004-walk.sh` returns HTTP 200, the walker parses the `/v1/ask` body and records the *quality* of the invited stranger's first-value, not merely that a 200 came back. `/v1/ask` returns one of two shapes (`apps/api/src/index.ts`): a `create` envelope (`{kind:"create", db, schemaName, engine, plan, sampleRows}` — emitted because an invited stranger has zero DBs, so the goal provisions one) or a `query` AskResult (`{status:"ok", rowCount, trace:{sql, confidence, model}}`). The walker detects which arrived and grades it: a `create` is `first_value_quality:"ok"` when it carries a real `db` + `schemaName` AND seeded ≥ 1 sample row across its tables (it also records `table_count` + `row_count` = distinct seeded tables + sample-row count + `engine`); a `query` is `ok` when `status=="ok"` AND the `trace.sql` is SELECT-backed (recording `rowCount`, `trace.confidence`, `trace.model`). Quality is **recorded, never fatal** — a 0-row query on a freshly provisioned DB is legitimate first-value, so a `degraded` grade annotates the outcome note and JSON but does not flip the walk's pass/fail state (that remains the SK-STRG-002 control×invite gate-bypass contract). The outcome JSON (canonical shape owned by [`SK-STRG-002`](../FEATURE.md)) gains `first_value_kind`, `result_status`, `row_count`, `table_count`, `answer_confidence`, `answer_model`, `sql_is_select`, `first_value_quality`. The `trace.sql` body is read only to classify SELECT-vs-not and is never logged (it can echo schema identifiers).

- **Core value:** Honest latency, Bullet-proof, Goal-first

- **Why:** FLOW-004 is the only canonical-five flow that carries a stranger *across* the gate to real first-value before BIRD/Spider clear ([`GLOBAL-027`](../../../decisions/GLOBAL-027-pre-alpha-gate.md) / [`GLOBAL-032`](../../../decisions/GLOBAL-032-top-5-user-flows-canonical.md)), so it is the one flow where "the invited stranger's experience is actually good" is a measurable, load-bearing claim — not a hypothetical behind a 403. Before this decision the walker asserted only `ask_status == 200`: a blank 200, an empty-schema provision, or a non-SELECT degenerate answer would all have counted as a clean pass, and the acquisition tracker would have claimed "invited first-value works" on evidence that proves only "the gate honoured the code." The 2026-06-07 verification run made the gap concrete — the first cut of this assertion assumed a `query` shape and graded the live response `degraded` (empty `status`), which surfaced that the real first-value is a *provisioned DB* (`kind:"create"`), not a query; the corrected walker then recorded `ok` with `6 tables / 13 sample rows / postgres`. That is the honest, surface-observable answer to "is their experience amazing once they cross the gate?" — and it is exactly the kind of claim [`GLOBAL-030`](../../../decisions/GLOBAL-030-evidence-grade-acquisition-tracker-edits.md) requires the tracker to back with a real artifact.

- **Consequence in code:** The walker's pass/fail decision matrix is unchanged (SK-STRG-002 still owns it); the quality parse runs only inside the existing `ASK_STATUS == 200` branch and adds no new exit code. The daily `acquisition-health.yml` cron summary surfaces `first_value_quality` + `first_value_kind` + `table_count` + `row_count` for the FLOW-004 step so a `degraded` grade is visible in the artifact without downloading it. Any future change to the `/v1/ask` success envelope (new top-level shape, renamed `sampleRows`/`trace`) must update the two `jq` shape branches here; a shape the walker can't classify lands as `first_value_kind:"unknown"` + `first_value_quality:"degraded"` (recorded, not crashed). When BIRD/Spider clear and the gate opens globally, the same quality fields apply to anon (non-invited) first-value — the assertion is not invite-specific, only its current trigger is.

- **Alternatives rejected:**
  - **Keep asserting only HTTP 200.** The status quo; it cannot distinguish "invited stranger got a seeded DB" from "invited stranger got a blank 200." That is precisely the shipped-≠-useful self-deception the acquisition tracker preamble exists to prevent.
  - **Fail the walk on `degraded`.** Would make a legitimate 0-row query (a valid answer on a fresh DB) a false regression and re-introduce flakiness into the daily cron — the opposite of the SK-STRG-002 "regression detector, not a liveness probe" property. Quality is a recorded signal the next agent reads, not a hard gate.
  - **Assert a specific row/table count threshold.** Brittle — the create-time schema size depends on the goal and the model; pinning "≥ 6 tables" would break the first time the planner emits a smaller-but-correct schema. "≥ 1 seeded row in a real schema" is the minimal honest bar for "the DB is usable."
  - **Re-query the new DB with a second `/v1/ask` to test the query path too.** Doubles the SK-GATE-007 invite cost per walk and couples FLOW-004 to FLOW-007 (anon-DB adoption / follow-up query). The create envelope already proves first-value; the query-path quality assertion fires for free whenever a walk does hit the query shape (e.g. a future seeded-DB variant).
  - **Log the generated SQL / sample rows for richer evidence.** Leaks schema identifiers (and potentially the goal) into the GH Actions raw log. The bounded counts (`table_count`, `row_count`, `confidence`) are the evidence; the SQL is classified and discarded, matching the SK-STRG-002 `redact()` contract.
