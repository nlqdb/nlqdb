# SK-STRG-003 — Daily GH Actions cron walks all walkers at 06:00 UTC; exits 0 unconditionally so no founder-facing email channel is created

- **Decision:** `.github/workflows/acquisition-health.yml` schedules `bash scripts/verify-flows.sh + bash scripts/stranger-test.sh + bash scripts/flow-005-walk.sh + bash scripts/flow-005-stdio-walk.sh` daily at `0 6 * * *` (same UTC slot as the weekly ICP scrape). Each step runs under `continue-on-error: true`; the job always reaches `upload-artifact` and exits success regardless of walker outcomes. Walker outputs (`verify-flows.log`, `walk-cron.json`, `flow-005-cron.json`, `flow-005-stdio-cron.json`) upload as `acquisition-health-<run_id>` with 90-day retention; the next agent fetches via `mcp__github__list_workflow_runs` + `actions/download-artifact`. Playwright Chromium is cached across runs via `actions/cache@v4`. `workflow_dispatch` accepts `base_url` (preview override for the marketing / app surfaces) and `mcp_url` (preview override for the MCP host the FLOW-005 walker hits).

- **Core value:** Bullet-proof, Simple, Goal-first

- **Why:** Closes the §1.1 daily cron + the [`GLOBAL-032`](../../../decisions/GLOBAL-032-top-5-user-flows-canonical.md) "every canonical-five walker ran within the last seven days" rule (daily run trivially satisfies it). A GH Actions runner satisfies §1.1's stranger-IP rule (a Worker IP would violate it); a CF cron Worker can't host Chromium anyway. The `exit-0-unconditionally` posture is load-bearing: GitHub emails the actor on scheduled-workflow failure on the default branch — that IS a founder-facing notification channel, which the operator loop ([`automated-icp-validation-plan.md`](../../../research/automated-icp-validation-plan.md) preamble + [`GLOBAL-028`](../../../decisions/GLOBAL-028-acquisition-progress-tracker.md)) forbids. Draining failure into the artifact JSON routes it back through the next agent's pick-list as required. Daily cadence: a handful of HTTP calls/day for FLOW-005 plus the Playwright walk — well inside [`GLOBAL-013`](../../../decisions/GLOBAL-013-free-tier-bundle-budget.md).

- **Consequence in code:** Adding a new agent-runnable walker means appending a `continue-on-error: true` step writing to `tools/stranger-test/results/<name>-cron.json` and a row + outcome block in the run-summary step. The workflow has no PR-side gate — `ci.yml::test-tools` continues to typecheck + unit-test `tools/stranger-test/`; the cron is production-surface only. The "always succeed" rule means a regression is invisible from email — the operator discovers it by running the agent prompt, by design. If a future requirement demands fail-loud on a specific walker, the right shape is a separate sink writing back to the impl plan's progress log via `mcp__github__create_or_update_file`, NOT changing this workflow's exit code.

- **Alternatives rejected:**
  - **Cloudflare cron Worker triggering the script remotely.** Can't host Chromium without `@cloudflare/playwright` (beta) or Browserless. Strictly more complex than a GH Actions runner that already has Bun + cached Chromium.
  - **Fail-loud on walker non-zero exit.** GitHub emails the repo owner on scheduled-workflow failure on default branch — operator-loop violation.
  - **One workflow per walker.** N artifact URLs, N summary tabs. One workflow shows the whole acquisition-health state in one place.
  - **Commit results JSON back into the repo via follow-up PR.** Adds CI→default-branch write path and bloats history forever. Artifacts are the right shelf-life for cron evidence.
  - **GitHub Issues notification on regression.** Issues email watchers — same operator-loop violation as fail-loud.
