# SK-STRG-007 — A FLOW-004 `create` first-value with 0 seeded rows records `state:"passed_degraded"`, not a bare `passed`

- **Decision:** When `scripts/flow-004-walk.sh` grades the invited stranger's first-value (per [`SK-STRG-006`](./SK-STRG-006-flow-004-first-value-quality.md)) and the `/v1/ask` response is a `create` envelope that provisioned a DB but seeded **0 sample rows** — the [`SK-HDC-018`](../../hosted-db-create/decisions/SK-HDC-018-sample-insert-graceful-degradation.md) un-seeded fallback, fired because the LLM's seed rows violated their own schema's FK/NOT-NULL — the walker records `state:"passed_degraded"` in the outcome JSON instead of a bare `passed`. The **process exit code is unchanged (0)**: the [`SK-STRG-002`](../FEATURE.md) control×invite gate-bypass invariant still passed (control 403 + invite 200), so the composer (`stranger-test-invited.sh`, which keys on the exit code to mint an invite) and the `acquisition-health.yml` cron (which keys on the exit code for its loop) are unaffected. Only the recorded `.state` — read by the §0.5 / status-dashboard / cron-summary tables for their at-a-glance column — carries the first-value verdict. A 0-row *query* keeps `state:"passed"` (a 0-row query on a fresh DB is legitimate first-value per SK-STRG-006); the distinction is that for a `create` the seeded rows **are** the first-value, so an empty DB is not first-value.

- **Core value:** Honest latency, Bullet-proof, Goal-first

- **Why:** SK-STRG-006 deliberately kept first-value quality "recorded, never fatal" so a legitimate 0-row query couldn't flip the daily cron to a false regression. That reasoning holds for queries but conflated the `create` case: the 2026-06-09 deployed re-walk proved a `create` can land `first_value_quality:"degraded"` with `state:"passed"`, so the dashboard's State column showed a flat "passed" for an invited stranger who actually got an **empty DB**. That is precisely the shipped-≠-useful self-deception the acquisition tracker's anti-self-deception layer (§1) exists to catch — the funnel reaching a 200 is not the same as the stranger getting a usable, seeded database. Carrying the verdict in `.state` (which the dashboards already read via `jq -r '.state'`) closes the gap without re-introducing the cron-flakiness SK-STRG-006 rejected, because the exit code — the only signal the composer/cron branch on — is untouched.

- **Consequence in code:** `scripts/flow-004-walk.sh` sets `WALK_STATE="passed_degraded"` in the `create` branch when `FIRST_VALUE_QUALITY` is `degraded`, and the exit-code `case` adds a `passed_degraded)` arm that prints a "first-value DEGRADED — un-seeded DB" line and `exit 0`. No other walker, the cron workflow, or the composer changes: `acquisition-health.yml` reads `.state` generically and now surfaces `passed_degraded` automatically; `stranger-test-invited.sh` proceeds on exit 0 as before. Any future change that makes the `create` envelope report seeded rows differently must keep the `ROW_COUNT ≥ 1` test (the bar for `ok`) in sync. The seed-quality root cause — making *every* goal's LLM seed rows insertable — is an engine-quality / [`SK-LLM-033`](../../llm-router/decisions/SK-LLM-033-schema-infer-insertable-sample-rows.md) lift tracked under the impl plan's preamble priority #1, not this walker's job.

- **Alternatives rejected:**
  - **Leave it as a bare `passed` (status quo).** The State column then shows "passed" for an empty-DB stranger — the exact self-deception §1 is built to prevent. The `first_value_quality:"degraded"` field exists but only in the per-flow detail block, not the at-a-glance table the next agent reads first.
  - **Make a degraded `create` exit non-zero (e.g. `partial`/exit 3).** Would abort `stranger-test-invited.sh` (it treats any non-zero `flow-004-walk.sh` exit as a mint failure, lines 86–93) even though the invite minted fine, and would flip a healthy gate-bypass invariant to a "failure" in the cron — re-introducing the flakiness SK-STRG-006 explicitly rejected. The gate-bypass invariant and the first-value quality are two different claims; only the latter is degraded.
  - **Also flip a degraded *query* to `passed_degraded`.** A 0-row query on a freshly provisioned DB is a valid answer; flagging it would re-create the false-regression problem SK-STRG-006 names. The refinement is `create`-specific on purpose.
  - **Add a typed enum for the state field validated per-PR.** Over-engineered for a free-form bash status string consumed only by human/agent-read dashboards and one generic `jq -r '.state'`. Promote if a programmatic consumer ever branches on the state string.
