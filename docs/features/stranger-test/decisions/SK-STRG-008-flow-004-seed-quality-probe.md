# SK-STRG-008 — FLOW-004 first-value seed-quality probe: many goals, a measured `seeded_ok_ratio`

- **Decision:** `scripts/flow-004-seed-quality.sh` measures what fraction of strangers get a *seeded* first-value DB rather than an empty one. It composes `scripts/flow-004-walk.sh` to set up, then issues a `create` ask per goal across a small P1 goal set directly against `/v1/ask`, each on a fresh `anon_<uuid>` bearer so every ask provisions a new DB. It grades each first-value with the same SK-STRG-006 rubric (`create` is `ok` only with a real `db` + `schemaName` and ≥1 seeded row) and writes `{seeded_ok_ratio, ok, degraded, provision_failed, errored, goals:[…]}` to `tools/stranger-test/results/flow-004-seed-quality-<utc>.json`. `seeded_ok_ratio` is computed over *classified* creates (ok + degraded); everything else is excluded so no non-seeding outcome masquerades as a seeding failure. The excluded set is split into two reported buckets: `provision_failed` (HTTP 422 — the engine couldn't build the DB at all; a *harder* first-value failure than `degraded`, surfaced separately so it stays visible rather than hidden among blips) and `errored` (true upstream blips — network failures, non-create 200s, other non-200s). It is a measurement, not a regression gate: it exits 0 whenever it produced a ratio (even 0/N), matching the SK-STRG-003 operator-loop philosophy. It is deliberately **not** in the daily `acquisition-health.yml` cron — each run provisions 1+N throwaway DBs, so it stays agent-on-demand to keep Neon free-tier pressure bounded.

- **Core value:** Bullet-proof, Honest latency, Goal-first

- **Why:** FLOW-004 carries a stranger from waitlist signup to real first-value, so its first-value quality *is* the acquisition funnel's first impression. SK-STRG-006/007 grade that quality for ONE goal per run, which made the tracker's "first-value seed quality is prompt-variable" claim an anecdote — two hand-picked prompts (a tiny CRM `ok`, a meal-planner `degraded`), never a measured rate. That under-measurement hid a regression: the 2026-06-10 first run measured `seeded_ok_ratio = 0.25` (1/4 — only "a habit tracker for my morning routine" seeded; "a tiny CRM" — previously recorded `ok` — degraded alongside the meal-planner and a reading-list goal). The probe turns the single most decision-relevant acquisition number (what fraction of strangers see a populated DB) into one cheap measurement, directly sizing the documented #1 engine-quality / SK-LLM-033 bottleneck instead of guessing from anecdote.

- **Consequence in code:** The probe re-uses every primitive already in `flow-004-walk.sh` (mint, redact, uniform JSON), so there is no new credential surface and no change to the daily cron's cost. Reading the seed-quality JSON is how the next agent sizes the engine-quality lift: a ratio well below 1.0 means the SK-LLM-033 insertable-seed-row prompt does not yet seed every goal, and the gap is engine-quality work (rolls up under the impl plan's priority #1), not a FLOW or walker gap. If the set ever grows past a handful of goals, or the probe needs to run on a schedule, promote it to its own throttled cron rather than folding it into `acquisition-health.yml` (which must stay cheap enough to run daily).

- **Alternatives rejected:**
  - **Run `flow-004-walk.sh` N times with different `FLOW_004_GOAL` values.** Mints N throwaway inboxes and N waitlist sends for N data points; the whole value here is N goals issued directly off one setup.
  - **Re-use one `anon_<uuid>` across all goals.** After the first `create` the principal owns a DB, so subsequent asks become `query` against it — not a comparable `create` per goal. A fresh anon bearer per goal is what keeps every measurement a create.
  - **Add it to the daily `acquisition-health.yml` cron.** Each run provisions 1+N throwaway DBs; daily that is steady Neon free-tier pressure for a number that moves only when engine-quality work lands. Agent-on-demand is the right cadence; the GLOBAL-032 freshness rule is about the canonical-five walkers, and FLOW-004's canonical walker (`flow-004-walk.sh`) is already in the cron.
  - **Fail (exit non-zero) when `seeded_ok_ratio < 1.0`.** Re-introduces a pass/fail gate on a metric that is expected to be below 1.0 until SK-LLM-033 lands — it would just be a permanently red check. The number is the deliverable; the engine-quality priority is where it gets acted on.
