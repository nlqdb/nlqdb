# SK-STRG-008 — FLOW-004 first-value seed-quality probe: one invite, many goals, a measured `seeded_ok_ratio`

- **Decision:** `scripts/flow-004-seed-quality.sh` measures what fraction of invited strangers get a *seeded* first-value DB rather than an empty one. It composes `scripts/flow-004-walk.sh` (via the `FLOW_004_INVITE_OUT` sidecar seam, exactly like the SK-STRG-004 composer) to mint and end-to-end-validate ONE invite (control 403 + invite 200), then re-uses that code — invite codes are validated by KV existence in [`apps/api/src/gate/bypass.ts`](../../../../apps/api/src/gate/bypass.ts) `isInviteValid`, not consumed, so one code drives many `/v1/ask` calls while the 200/week cap in [`waitlist-invite.ts`](../../../../apps/api/src/waitlist-invite.ts) only meters issuance — to issue a `create` ask per goal across a small P1 goal set, each on a fresh `anon_<uuid>` bearer so every ask provisions a new DB. It grades each first-value with the same SK-STRG-006 rubric (`create` is `ok` only with a real `db` + `schemaName` and ≥1 seeded row) and writes `{seeded_ok_ratio, ok, degraded, errored, goals:[…]}` to `tools/stranger-test/results/flow-004-seed-quality-<utc>.json`. `seeded_ok_ratio` is computed over *classified* creates (ok + degraded); errored goals (network / non-200) are excluded so an upstream blip can't masquerade as a seeding failure. It is a measurement, not a regression gate: it exits 0 whenever it produced a ratio (even 0/N), matching the SK-STRG-003 operator-loop philosophy. It is deliberately **not** in the daily `acquisition-health.yml` cron — each run provisions 1+N throwaway DBs, so it stays agent-on-demand to keep Neon free-tier pressure bounded.

- **Core value:** Bullet-proof, Honest latency, Goal-first

- **Why:** FLOW-004 is the only path that carries a stranger across the GLOBAL-027 gate before BIRD/Spider clear, so its first-value quality *is* the acquisition funnel's first impression. SK-STRG-006/007 grade that quality for ONE goal per run, which made the tracker's "first-value seed quality is prompt-variable" claim an anecdote — two hand-picked prompts (a tiny CRM `ok`, a meal-planner `degraded`), never a measured rate, and each extra data point cost another invite. That under-measurement hid a regression: the 2026-06-10 first run measured `seeded_ok_ratio = 0.25` (1/4 — only "a habit tracker for my morning routine" seeded; "a tiny CRM" — previously recorded `ok` — degraded alongside the meal-planner and a reading-list goal). The probe turns the single most decision-relevant acquisition number (what fraction of invited strangers see a populated DB) into one cheap measurement, directly sizing the documented #1 engine-quality / SK-LLM-033 bottleneck instead of guessing from anecdote.

- **Consequence in code:** The probe re-uses every primitive already in `flow-004-walk.sh` (mint, redact, uniform JSON, control×invite invariant) and the SK-STRG-004 sidecar hand-off, so there is no new credential surface and no change to the daily cron's cost. Reading the seed-quality JSON is how the next agent sizes the engine-quality lift: a ratio well below 1.0 means the SK-LLM-033 insertable-seed-row prompt does not yet seed every goal, and the gap is engine-quality work (rolls up under the impl plan's priority #1), not a FLOW or walker gap. If the set ever grows past a handful of goals, or the probe needs to run on a schedule, promote it to its own throttled cron rather than folding it into `acquisition-health.yml` (which must stay cheap enough to run daily).

- **Alternatives rejected:**
  - **Run `flow-004-walk.sh` N times with different `FLOW_004_GOAL` values.** Burns N invites of the 200/week cap for N data points; the whole value here is N goals on ONE invite (codes are existence-checked, not consumed).
  - **Re-use one `anon_<uuid>` across all goals.** After the first `create` the principal owns a DB, so subsequent asks become `query` against it — not a comparable `create` per goal. A fresh anon bearer per goal is what keeps every measurement a create.
  - **Add it to the daily `acquisition-health.yml` cron.** Each run provisions 1+N throwaway DBs; daily that is steady Neon free-tier pressure for a number that moves only when engine-quality work lands. Agent-on-demand is the right cadence; the GLOBAL-032 freshness rule is about the canonical-five walkers, and FLOW-004's canonical walker (`flow-004-walk.sh`) is already in the cron.
  - **Fail (exit non-zero) when `seeded_ok_ratio < 1.0`.** Re-introduces a pass/fail gate on a metric that is expected to be below 1.0 until SK-LLM-033 lands — it would just be a permanently red check. The number is the deliverable; the engine-quality priority is where it gets acted on.
