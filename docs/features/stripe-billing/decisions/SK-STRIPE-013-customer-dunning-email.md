# SK-STRIPE-013 — Customer dunning email rides the existing `billing.payment_failed` event into a Resend sink

Parent feature: [`stripe-billing/FEATURE.md`](../FEATURE.md). This is the
customer-facing half of dunning; the operator alert is `SK-STRIPE-011`, the
in-app banner is web-app `SK-WEB-012`, and the sink lives in the
events-pipeline feature.

- **Decision:** The customer-facing payment-failure reminder is sent from the **events-worker**, not the webhook handler. The `billing.payment_failed` event (already emitted for the operator alert, SK-STRIPE-011) carries `customerEmail` (from `invoice.customer_email`); a `dunning-email.ts` sink builds the message body and the worker sends it through the `@nlqdb/email` Resend owner with an `Idempotency-Key` of the per-invoice envelope id. The send is **best-effort**: it runs after the ack-deciding LogSnag publish and swallows its own errors, so a Resend outage never retries the message and re-pages the operator. Inert until `RESEND_API_KEY` is set and a real `invoice.payment_failed` fires (same dev/unconfigured no-op posture as the LogSnag + Tinybird sinks).
- **Core value:** Honest latency, Bullet-proof, Simple
- **Why:** Recovering involuntary churn means reaching the *customer* before Stripe exhausts its ~2-week retry schedule — the operator alert (SK-STRIPE-011) and in-app banner (SK-WEB-012) only help if the customer is already looking. The events-worker is the right home because per SK-EVENTS-001 it is the only surface that talks to external sinks (the rejected-alternative there is exactly an inline `apps/api` `ctx.waitUntil` send): a Resend round-trip belongs off the webhook's response path, beside the LogSnag alert this same event already drives, keeping the webhook's 200 fast. The transport is the existing `@nlqdb/email` Resend owner (GLOBAL-021) — also used by magic-link — so there's no new vendor, no second Resend wire surface, no extra Stripe/DB read, and the go-live flip is config-only (the ship-inert-early stance of SK-STRIPE-008).
- **Consequence in code:** `customerEmail` added to the `billing.payment_failed` variant in `packages/events/src/types.ts` and populated in `handleInvoicePaymentFailed`; the Resend transport moves to the new `@nlqdb/email` package (GLOBAL-021 owner row; `EmailMessage` gains an optional `idempotencyKey`); `apps/events-worker/src/sinks/dunning-email.ts` builds the message body and `apps/events-worker/src/index.ts` sends it via `makeEmailSender` under a `nlqdb.billing.dunning_email` span (GLOBAL-014); `RESEND_API_KEY`/`RESEND_FROM` added to the events-worker env + the worker's secret subset in `scripts/mirror-secrets-workers.sh`. Resend's 24h idempotency window collapses Cloudflare Queue redeliveries; Stripe's spaced retries (~4 over two weeks) legitimately each remind the customer (ordinary dunning, not spam).
- **Alternatives rejected:**
  - Send synchronously from the `invoice.payment_failed` webhook handler — puts an 8 s Resend round-trip on the webhook's response path, the exact hot-path coupling the events pipeline exists to avoid; Stripe wants a fast 2xx.
  - A second Stripe/D1 lookup for the email in the worker — `invoice.customer_email` is already on the event source; riding it on the event is one read fewer and keeps the worker sink-pure.
  - Let the email failure retry the message — re-pages the operator (LogSnag dedups, but the retry budget burns) for a best-effort comm; the operator alert is the source of truth.
