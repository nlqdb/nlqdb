# SK-STRIPE-012 — The webhook pipeline is order-independent: `customer.subscription.created` self-heals the link from subscription metadata

Parent feature: [`stripe-billing/FEATURE.md`](../FEATURE.md). The user id this
relies on is stamped at checkout per `SK-STRIPE-004`; the row creation it
shares lives in `SK-STRIPE-004`'s `checkout.session.completed` handler.

- **Decision:** `handleSubscriptionCreated` resolves the user by the `stripe_customer_id` lookup first; on a miss (no `customers` row yet) it falls back to `subscription_data.metadata.nlqdb_user_id` (SK-STRIPE-004) and creates the row via the same idempotent `upsertCustomerLink` `checkout.session.completed` uses, then syncs fields and emits `billing.subscription_created`. Whichever of the two events arrives first creates the row; the other upserts. Absent metadata (out-of-band sub) → warn `subscription_created_no_user_mapping` and skip. `updated`/`deleted` keep the lookup-only `resolveSubUser` — they presuppose a prior `created`.
- **Core value:** Bullet-proof
- **Why:** Stripe explicitly does not guarantee webhook delivery order ([docs](https://docs.stripe.com/webhooks)), so `customer.subscription.created` can land before `checkout.session.completed`. The old lookup-only path then dropped the `billing.subscription_created` operator alert permanently and left the row `incomplete` until an unrelated later `updated` happened by. Carrying the user id on the subscription itself makes each event self-sufficient — the recommended Stripe pattern over depending on order or issuing a reconciling API read.
- **Consequence in code:** `upsertCustomerLink` is the one INSERT…ON CONFLICT shared by `handleCheckoutCompleted` and the fallback; `readUserIdFromMetadata` reads the stamped id. The ON CONFLICT touches only the Stripe ids (not `status`), so a late `checkout.session.completed` can't downgrade a status the subscription already synced — robust under both out-of-order and concurrent (parallel-Worker) delivery, since `syncSubscriptionFields` is the only `status` writer and each step is independently idempotent. The second unique constraint, `customers.stripe_customer_id` (migration 0004), is unreachable on the normal flow (checkout mints a fresh Stripe customer per session); an out-of-band Dashboard collision throws and fails safe to the existing stuck-event path (`processed_at` stays NULL for replay). No new Stripe call, no new env var, no schema change. Unit-tested both ways (row-present normal path; empty-DB metadata self-heal).
- **Alternatives rejected:**
  - Reconcile via a Stripe API read when the row is missing — couples the webhook to an outbound call (latency + a Workers-egress dependency) for data we already hold at checkout.
  - Leave the skip-and-backfill behaviour — a paid customer silently stays unlinked until manual operator action; contradicts the "Bullet-proof" core value.
