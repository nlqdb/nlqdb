# SK-ICP-014 — Empty runs write a starvation-marked evidence file + a `starved` signal

- **Decision:** When `runIcpCluster` finds an empty scored set (no `icp:scored:*` keys, or all values aged past their 30-day TTL between `list` and `get`), it no longer silently early-returns `written:false`. It writes the monthly `docs/research/icp-evidence-<yyyy-mm>.md` anyway, containing a `## Starvation notice` section: an explicit zero-item statement, the most recent scrape's per-source new-item counts, and which sources self-skipped for missing env keys. To feed that section, `runIcpScrape` persists a single `icp:last_scrape_stats` KV record (`{ ts, newItems, skipped, sources, skippedSources }`, 90-day TTL) on every run; `runIcpCluster` reads it on the starved path. The result gains `starved?: boolean`, which flows into the existing `icp_cluster_completed` console log (via the `...clusterResult` spread in `index.ts`) and into the LogSnag description, so a log-based alert can distinguish "healthy-quiet" from "starved". No new OTel span attribute names are introduced — the existing `nlqdb.icp.github_write` span now fires on the empty path too (previously it did not). **Scoring, the relevance floor, and the source set are unchanged** — those remain parked (Reddit off per SK-ICP-011; Stack Exchange parked); this decision only makes the zero legible.

- **Core value:** Bullet-proof, Honest latency

- **Why:** The lone May 2026 scored item expired ~2026-06-23 under the 30-day TTL. From then, every weekly cron produced nothing and the old early-return wrote no file and (before this) gave the founder no signal — the acquisition-research pipeline flatlined for 7+ weeks with no visible trace. Silent zero is the worst failure mode for a research pipeline: it is indistinguishable from "no cron ran". Writing a starvation-marked file each run turns an invisible drought into a dated artifact the founder and the daily agent read, and the `starved:true` log field lets a cheap log-based alert fire without a new metric series (GLOBAL-014 / observability SK-OBS-002 cardinality budget).

- **Consequence in code:** `apps/api/src/icp-scrape.ts` gains `LAST_SCRAPE_STATS_KEY`, `IcpScrapeStats` (exported), and a `deps.kv.put(LAST_SCRAPE_STATS_KEY, …)` before the LogSnag call; `skippedSources` is derived from env-key presence (`reddit` when Reddit creds absent; `github` + `github_discussions` when `GH_TOKEN` absent). `apps/api/src/icp-cluster.ts` imports both, adds `starved?` to `IcpClusterResult`, extracts the shared `writeEvidenceFile` helper (the `nlqdb.icp.github_write` span used by both the normal and starvation paths), adds `readLastScrapeStats` + `generateStarvationMarkdown`, and replaces the two empty-set early-returns with one starvation branch. `notifyLogSnag` emits a starvation-specific description when `result.starved`. `index.ts` is unchanged — the `starved` field rides the existing `...clusterResult` log spread. Tests: `icp-cluster.test.ts` covers empty-set → starvation file with per-source counts, absent-stats fallback wording, and the list-returns-keys-but-values-gone TTL case; `icp-scrape.test.ts` covers the `icp:last_scrape_stats` round-trip (sources + self-skipped sources).

- **Alternatives rejected:** Keep the silent early-return (the defect — 7 weeks invisible). A dedicated LogSnag "starved" channel-bell event (over-engineered vs a log field; the parked threshold-alert open question already governs when a bell is warranted). A separate `icp:starvation` marker file distinct from the monthly evidence file (fragments the founder's one reading surface; the monthly file is where they already look). Lowering the relevance floor or re-enabling Reddit to manufacture signal (would contradict SK-ICP-002/010/011 — parked decisions, out of scope; P1). A new OTel span attribute for `starved` (would need a `docs/performance.md` §3 catalog row per SK-OBS-001; the console-log field carries the same signal for free).
