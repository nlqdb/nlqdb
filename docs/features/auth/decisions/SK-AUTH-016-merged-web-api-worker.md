# SK-AUTH-016 — `apps/web` and `apps/api` ship as a single Cloudflare Worker on `app.nlqdb.com`

- **Decision:** The product UI (`/app/*`, `/auth/*`, the sign-in island) and the HTTP API (`/api/auth/*`, `/v1/*`) are served by the same Cloudflare Worker — `nlqdb-api` — at `app.nlqdb.com`. Static Astro output from `apps/web/dist` is wired in via the worker's `[assets]` binding (`apps/api/wrangler.toml`); the platform serves any request whose path matches a built file directly, and falls through to the Hono handler for everything else (no code change required in `src/index.ts` to compose the priority ordering). The marketing surface remains a separate worker (`nlqdb-web`) at `nlqdb.com`, serving the same Astro build for marketing pages only. `SK-AUTH-013`'s preview-URL CORS / `trustedOrigins` widenings are removed for production — the cross-eTLD+1 condition that motivated them no longer applies once UI + API share an origin.
- **Core value:** Bullet-proof, Seamless auth, Effortless UX
- **Why:** Two production bugs traced back to the cross-origin web/API split: (a) the preview-branch 401 — Workers-Versions previews of `apps/web` landed on a third eTLD+1 (`*-nlqdb-web.omer-hochman.workers.dev`) that was outside the cookie's `Domain=.nlqdb.com` scope, so the cookie was dropped on every API call; and (b) the `SK-AUTH-015` OAuth-state-cookie loss on the same cross-eTLD+1 flow. Both are dialect bugs of "the cookie has to travel cross-origin", and the trend (browser cookie partitioning) is widening, not narrowing. Collapsing the product surfaces onto one origin removes the travel: every product-side request is first-party, the cookie can be host-only (no `Domain=` attribute, see `SK-WEB-009`), and the architectural class of bugs disappears for prod. The marketing surface keeps its own worker because (i) it has its own SEO concerns documented in `apps/web/wrangler.toml`, (ii) anonymous-bearer calls from `nlqdb.com` to `app.nlqdb.com/v1/ask` work cross-origin without cookies, and (iii) no marketing surface needs an authed cookie.
- **Consequence in code:** `apps/api/wrangler.toml` carries `[assets] directory = "../web/dist"`. CI builds `apps/web` before deploying `apps/api` (otherwise the directory is missing — see `.github/workflows/deploy-api.yml`). `apps/api/src/index.ts` trims preview-URL regexes from `CORS_ALLOWED_ORIGINS`. `apps/api/src/auth.ts` trims preview wildcards from `trustedOrigins`. `buildSignInUrl` in `apps/api/src/index.ts` anchors on `auth.options.baseURL` (= the merged origin) and points at `/auth/sign-in` (the actual sign-in route, not the historical `/sign-in`). `MAGIC_LINK_WEB_ORIGIN` defaults to `https://app.nlqdb.com` so the magic-link continue page lands on the merged origin. PRs that re-introduce a separate worker for the product UI are rejected.
- **Alternatives rejected:**
  - **Keep two workers and rely on `Domain=.nlqdb.com` cookie indefinitely.** Browser cookie partitioning is widening; the same auth-bug debt would resurface every time a new browser ships a new third-party-cookie posture.
  - **Single worker for marketing + product (collapse `apps/web` entirely).** Conflates SEO concerns (marketing's `workers_dev = false` matters for indexing) and forces every marketing static-assets change through the API deploy pipeline. Two workers, one shared build, is the smaller blast radius.
  - **Split `apps/web` into two Astro builds (one for marketing, one for product).** Cleaner conceptually but adds a second config + CI matrix entry; deferred until a concrete reason to disjoin the two surfaces appears (e.g., divergent base layouts, divergent islands, divergent CSP).
  - **Stand up a per-PR preview API alongside each preview web (the rejected alternative in `SK-AUTH-013`).** OAuth Apps support exactly one callback URL (`SK-AUTH-008`); per-PR previews would each need their own GitHub OAuth App or a callback-URL-rewrite proxy. Too much surface area for the bug it solves.
