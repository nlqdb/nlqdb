# SK-AUTH-017 — Canary worker is the real-IdP integration gate between PR previews (mock) and production

- **Decision:** Maintain one persistent non-prod Cloudflare Worker (`nlqdb-api-canary`, config `apps/api/wrangler.canary.toml`) with real GitHub + Google OAuth Apps (the third pair from `SK-AUTH-008`), Stripe **test-mode**, a dedicated Neon branch, and its own KV / D1 / Queue / R2 bindings. The canary worker is deployed on every push to `main` (`.github/workflows/deploy-canary.yml`) and on demand via `workflow_dispatch`. It serves as the integration gate between PR previews (mock IdP, `SK-AUTH-018`) and production: real-IdP regressions surface here before customers see them.
- **Core value:** Bullet-proof, Honest latency, Seamless auth
- **Why:** PR previews use a mock IdP because per-PR real-OAuth Apps are infeasible (`SK-AUTH-008` — GitHub OAuth Apps support exactly one callback URL). Mock previews catch UI / API / cookie-attribute regressions but cannot exercise real provider callback semantics, real ID-token shapes, or real consent-screen quirks. Prod-only verification of those is too late. One persistent canary worker — same code, same secrets shape, real IdPs, isolated data — is the smallest delta that closes the gap. Stripe test-mode (not live mode) makes canary safe for billing-flow exercise without touching real customer money. A dedicated Neon branch keeps schema migration shape exercises off prod.
- **Consequence in code:** `apps/api/wrangler.canary.toml` enumerates canary bindings (placeholder IDs are filled in once `wrangler kv namespace create` / `d1 create` / `queues create` / `r2 bucket create` run for the canary names). `apps/api/src/auth.ts` reads `BETTER_AUTH_URL` (canary points at the canary `*.workers.dev` URL) and includes `CANARY_ORIGIN` in `trustedOrigins` when set. The `isDev` gate is `NODE_ENV !== "production" && NODE_ENV !== "canary"` so canary takes the prod-equivalent code path (cookie `__Secure` prefix, prod `OAUTH_GITHUB_*` slot — each worker has its own secret store). `apps/api/package.json` exposes `deploy:canary` and `secrets:canary` scripts. PRs that special-case canary in handler code (instead of leaving it as a binding-difference deployment) are rejected — canary's value is exercising the same code path with different bindings, not a parallel branch.
- **Alternatives rejected:** **Test only on prod traffic.** Real-IdP regressions only surface after deploy, paging the on-call. Canary moves them one stage earlier at near-zero ongoing cost. **Test only with mock previews (`SK-AUTH-018`).** Misses provider-side changes (Google consent-screen redesigns, GitHub scope deprecations, ID-token claim shifts). **Per-PR real-OAuth Apps.** Forbidden by `SK-AUTH-008` — single-callback constraint plus the registration overhead per PR. **Run canary as a Workers Versions preview of the prod worker.** Conflates test-mode billing + isolated D1 with a "non-promoted version" of prod; one wrong promote click ships test-mode Stripe to live customers. Better to keep them as separate workers. **Spin canary up per-deploy then tear it down.** Defeats the purpose of a stable callback URL — every spin re-registers OAuth Apps, which negates the only thing canary proves.
