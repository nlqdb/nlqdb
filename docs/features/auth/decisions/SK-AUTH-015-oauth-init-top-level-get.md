# SK-AUTH-015 ‚ÄĒ OAuth init is a top-level GET to `/api/auth/oauth-init/:provider`, not a `fetch + JS redirect`

- **Decision:** The sign-in page kicks off Google/GitHub OAuth by setting `location.href` to a server-rendered redirect endpoint (`GET /api/auth/oauth-init/:provider?callbackURL=‚Ä¶`) on the API. The endpoint wraps Better Auth's `POST /sign-in/social` server-side and replies with a 302 to the IdP. The previous flow ‚ÄĒ `fetch("‚Ä¶/sign-in/social", { credentials: "include" }) ‚Üí location.href = body.url` ‚ÄĒ is removed.
- **Core value:** Bullet-proof, Seamless auth, Effortless UX
- **Why:** `SK-AUTH-013` added preview-URL coverage to CORS + `trustedOrigins` so previews could complete OAuth. CORS is a request-time gate; what we missed is that the POST response *also* carries the OAuth state cookie. When the sign-in page is on `*.workers.dev` and the API is on `nlqdb.com` (Workers-Versions previews per `SK-AUTH-013` land at `<short-version-id>-nlqdb-web.omer-hochman.workers.dev`, while the API stays on `app.nlqdb.com`), the cross-eTLD+1 `fetch + credentials: include` is a third-party request from the browser's perspective. Chrome's third-party-cookie phase-out (default-on across stable Chromium since 2024) and Safari ITP both silently drop the `Set-Cookie` for the state cookie even when `SameSite=None; Secure` ‚ÄĒ the cookie attribute set is irrelevant once the partition rule applies. The OAuth callback then runs `parseGenericState`, finds no state cookie, and Better Auth redirects to `${baseURL}/error?error=state_mismatch` (`node_modules/better-auth/dist/oauth2/state.mjs:39`, `node_modules/better-auth/dist/state.mjs:105`). A top-level navigation makes `app.nlqdb.com` the first-party origin for the response that sets the cookie, so the cookie lands and the callback verifies cleanly. Same code path covers production (sign-in on `nlqdb.com`, API on `app.nlqdb.com` ‚ÄĒ same eTLD+1, top-level nav still works) and dev (`localhost`).
- **Consequence in code:** `apps/api/src/index.ts` registers `GET /api/auth/oauth-init/:provider` ahead of the Better Auth catch-all. The handler validates `provider ‚ąą {google, github}`, builds an internal `POST /api/auth/sign-in/social { provider, callbackURL }` `Request`, runs it through `auth.handler`, then forwards every `Set-Cookie` header along with the OAuth URL as a real 302. `apps/web/src/pages/auth/sign-in.astro` sets `location.href` instead of POSTing; OAuth-init URL + `callbackURL` anchor on a runtime-resolved `apiBase` (= `location.origin` on the merged app worker / previews / `wrangler dev`; = `https://app.nlqdb.com` when served from the marketing origin, which shares `apps/web/dist` per SK-AUTH-016 but lacks `/api/auth/*`). Telemetry mirrors the existing `nlqdb.auth.*` envelope: span `nlqdb.auth.oauth.init` + counter `nlqdb.auth.events.total{type=oauth_init, outcome}`. PRs that revive the `fetch + JS redirect` shape are rejected at review. On successful sign-in (magic-link verify or OAuth callback), the Better Auth `after` middleware (`apps/api/src/auth.ts`) reads the `__Secure-anon-bearer` cookie stashed via `POST /api/auth/anon-stash` and calls `recordAnonAdoption()` per SK-ANON-012 ‚ÄĒ span `nlqdb.anon.adopt` captures the outcome.
- **Alternatives rejected:** **Set `SameSite=None; Secure` on the state cookie and keep the fetch flow.** The third-party-cookie phase-out blocks partitioned cookies regardless of `SameSite`; this is a 2022 fix for a 2024 problem. **Use a hidden form POST as a top-level navigation.** Better Auth's `/sign-in/social` answers 200 with `{ url }` and a `Location` header, not a 302 ‚ÄĒ browsers don't follow `Location` on a 200, so the user would see raw JSON. Wrapping it server-side is the only way to get the redirect status. **Bounce through the production sign-in page (`nlqdb.com`) before initiating from preview.** Adds an extra navigation, doesn't preserve the preview origin in `callbackURL` without query-string round-tripping, and silently routes preview testers through prod state ‚ÄĒ defeats the point of a preview. **Re-enable cookies via the Storage Access API.** Requires a user gesture + per-site permission prompt; UX worse than a redirect. **Stand up a per-PR preview API on the same eTLD+1 as the preview web.** Already rejected in `SK-AUTH-013` (single-callback OAuth Apps + binding duplication).
