# SK-AUTH-018 — Preview-only `MOCK_IDP` / `MOCK_STRIPE` flags bypass external IdP, Resend, and Stripe round-trips

- **Decision:** Three env-flag-gated escape hatches let PR previews exercise the full sign-in → authenticated API call → billing-event loop without contacting any external provider:
  - `MOCK_IDP=1` activates `GET /auth/sign-in` (a one-click form) and `GET /api/auth/mock-sign-in` (handler), which trigger Better Auth's real magic-link plugin server-side: the magic-link `sendMagicLink` callback (gated on the same flag) sinks the URL to KV instead of calling Resend; the handler reads the URL back from KV and `auth.handler`'s the verify request to mint the cookie. The cookie is the same `__Secure-better-auth.session_token` that a real OAuth callback or magic-link click would set. `GET /api/dev/inbox` (also gated) returns the captured emails so synthetic clients can drive the loop programmatically.
  - `MOCK_STRIPE=1` flips `processWebhook` into bypass mode: `Stripe.webhooks.constructEventAsync` is skipped and the raw body is `JSON.parse`'d as the `Stripe.Event` instead. The idempotency insert, dispatch switch, R2 archive, and downstream `billing.*` emit run identically to the real signature-verified path. `STRIPE_WEBHOOK_SECRET` is no longer required when the flag is set (the `secret_unconfigured` 503 is suppressed).
  - Both flags MUST remain unset in production. CI sets them only on preview workers (`apps/api` Workers-Versions deployments). Canary (`SK-AUTH-017`) does NOT set them either — canary exists to exercise real IdPs. Every other auth surface — KV revocation, `cookieCache`, `requireSession`, `/v1/*` gating, the per-IP / per-email throttles, `trustedOrigins`, the `originCheck` middleware — runs real on previews.
- **Core value:** Effortless UX, Free, Bullet-proof
- **Why:** PR previews need to verify the full auth → session → API path on every change. Real OAuth Apps support exactly one callback URL (`SK-AUTH-008`); per-PR previews can't pre-register a callback per branch without rotating `n` apps. Magic-link goes to Resend, which (a) needs a verified sender domain and (b) doesn't deliver instantly to every preview test box. Stripe webhook signing requires the live HMAC secret + the dashboard endpoint configured against the preview URL — also impractical per-PR. The three flags collapse the external dependencies into in-process code paths while leaving the first-party auth + billing machinery wired exactly the same way it ships to prod, so a preview test that mints a session, calls `/v1/ask`, and processes a synthetic `customer.subscription.created` is a meaningful regression signal for everything except the third-party integration itself. Real-IdP regressions are caught one stage later by canary (`SK-AUTH-017`).
- **Consequence in code:** `apps/api/src/auth/mock-idp.ts` and `apps/api/src/auth/mock-email-sink.ts` are imported only by `apps/api/src/index.ts`'s `if (env.MOCK_IDP === "1")` block and by `apps/api/src/auth.ts`'s `sendMagicLink` callback (also `if (env.MOCK_IDP === "1")`). `apps/api/src/stripe/webhook.ts`'s `StripeWebhookDeps.bypassSignatureVerification` is set by `apps/api/src/index.ts` when `env.MOCK_STRIPE === "1"`. PRs that introduce `MOCK_IDP` / `MOCK_STRIPE` into `apps/api/wrangler.toml`'s `[vars]` block (i.e. into the prod-bound config rather than per-PR `wrangler versions upload --var`) are rejected at review. The mock sign-in form deliberately overrides any static `/auth/sign-in/index.html` served by the `[assets]` binding when active.
- **Alternatives rejected:** **Per-PR OAuth App (one app + one callback per preview).** GitHub OAuth Apps cap at "one callback URL per app"; managing one app per PR means rotating `n` client secrets and cleaning them up post-merge. Operational overhead far exceeds the cost of a flag. **Canary-only auth testing (no preview-level coverage).** Misses regressions until merge; the bug class we're protecting against (`SK-AUTH-013`/`SK-AUTH-015` cross-eTLD+1 cookie issues) is exactly what previews exist to catch. **Sandbox-mode Stripe + a per-PR webhook endpoint.** Stripe sandbox webhook endpoints are configured in the dashboard against a fixed URL — same per-PR-cleanup problem as OAuth Apps. **Reuse production keys + skip signature checks via a header (`X-Mock: 1`).** A header-driven bypass is data-dependent (any inbound request can set it); an env-flag-driven bypass is configuration-dependent (only set by infra). Configuration is the right granularity for "this entire worker version is preview-only." **Mock the entire Better Auth singleton in tests.** Loses fidelity — the bug class we're testing against (cookie-shape, KV-revocation timing, `originCheck` rejection) is in Better Auth itself; mocking it makes the test a tautology.
