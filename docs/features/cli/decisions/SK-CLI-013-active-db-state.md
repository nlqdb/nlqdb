# SK-CLI-013 — Active DB lives in `~/.config/nlqdb/state.json`; no `nlq init`, no project-level config in v1

- **Decision:** The CLI tracks the active DB (and any other mutating non-secret state — update-check timestamp, etc.) in `~/.config/nlqdb/state.json`. The file is JSON, not TOML, and lives separately from `config.toml` (prefs) and the keychain (secrets). No `nlq init` verb in v1. Project-level pinning (`.nlqdb/config.toml` in cwd) is explicitly deferred; a future `SK-CLI-NNN` revisits it if user demand surfaces (e.g. a recurring request from power users juggling multiple projects).
- **Core value:** Simple, Effortless UX
- **Why:** Per P5, drop machinery that v1 doesn't need. A state file solves the activation moment (`nlq new` then `nlq "..."` reuses the new DB) without forcing a project model. The earlier `cli/FEATURE.md` open question noted DISPATCH §14 listed `init` but DESIGN §3.3's happy-path doesn't use it; this resolves the contradiction by following the happy-path. Splitting state from prefs from secrets keeps [`SK-CLI-010`](SK-CLI-010-prefs-vs-secrets.md)'s "share my config without leaking" property intact: `state.json` lists `active_db` (not a secret, but volatile), `config.toml` lists prefs (stable, shareable), keychain holds secrets.
- **Consequence in code:** `cli/internal/state/` is the only module that reads/writes `state.json`. The schema is `{"active_db": "...", "last_used_at": "...", "update_check": {...}}`. Corruption recovery is "delete the file" — the next `nlq new` rewrites it. Reviewers reject `nlq init` PRs against this slice; [`SK-CLI-003`](SK-CLI-003-subcommand-verbs.md)'s canonical verb list (`ask`, `run`, `new`, `chat`, `db`, `keys`, `login`, `logout`, `whoami`, `use`, `mcp install`, `update`, `export`, `connection`) has no `init`, and Cobra's verb registry is the runtime mirror of that list.
- **Alternatives rejected:**
  - `nlq init` writing `.nlqdb/config.toml` per-project — extra mental model for v1; `gh repo set-default`-style project pinning matters once you have many projects, and we have no evidence v1 users do. Defer rather than build speculatively.
  - Bundle `active_db` into `config.toml` — mixes mutating state (changes on every `nlq use`) with stable prefs; a "share my config" copy now carries the live active DB. Defeats the [`SK-CLI-010`](SK-CLI-010-prefs-vs-secrets.md) safety property.
  - Store `active_db` in the keychain — adds keychain churn for non-secret data; turns `nlq use` into a keychain write on every call.
