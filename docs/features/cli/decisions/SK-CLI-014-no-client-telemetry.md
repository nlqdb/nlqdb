# SK-CLI-014 — No client-side telemetry pipeline; events ride the SDK's API calls

- **Decision:** The CLI does not implement a client-side telemetry event pipeline. All product telemetry (`feature.requested.*`, `home.surface_*`, the `GLOBAL-024` demand-signal events) emits server-side from `apps/api` once the CLI's HTTP request lands. The CLI sets `User-Agent: nlq/<version> (<install-method>; <os>/<arch>)` so the server can derive version + install method + platform from the access log without any client-emitted event. Offline-only CLI invocations (`nlq mcp install` writing host configs, `nlq logout` clearing the keychain) emit no telemetry at all.
- **Core value:** Bullet-proof, Open source, Free
- **Why:** A client-side telemetry pipeline is a compliance surface (disclosure copy, opt-out env vars, config knobs, CI detection, first-run prompt) *and* a reputational one — `gh` shipped opt-out telemetry on by default in April 2026 and was met with sustained backlash and a same-week opt-out changelog entry (see WebSearch 2026-05 receipts). Avoiding the pipeline entirely sidesteps both. The signals we actually need for `GLOBAL-024` (CLI version, install method, error codes, "not yet" surfaces) all land server-side on the existing `apps/api` event sink; the SDK and CLI just talk to the API. `User-Agent` parsing is one stable string format, not a new event schema. Open source: any user can run `nlq --version` and inspect the binary offline; there are no event endpoints to discover or block.
- **Consequence in code:** `packages/sdk` sets the `User-Agent` header on every request — TypeScript and the future Go SDK both build the string from `{name, version, install_method, platform}`. `cli/internal/version` is the single source of `install_method`, embedded at build time per install path (`curl-sh`, `homebrew`, `npm-shim`). `apps/api/src/principal.ts` parses `User-Agent`, tags the existing `cli.*` events with `cli_version` + `cli_install_method`. PRs that add a client-side `events` package under `cli/` or an HTTP POST to anything other than `api.nlqdb.com` (or the update-check endpoint per [`SK-CLI-015`](SK-CLI-015-update-check.md)) fail review. The CLI does **not** read `DO_NOT_TRACK`, `GH_TELEMETRY`, `NLQDB_TELEMETRY`, etc. — there is nothing to opt out of.
- **Alternatives rejected:**
  - Default-on opt-out with disclosure (gh's model) — invites the same backlash; adds env-var precedence, CI detection, first-run disclosure copy, `config.toml` flag, all to capture data that's already derivable server-side.
  - Opt-in prompt on first run — interactive prompt breaks `--json` consumers and CI; we don't actually need the data once `User-Agent` parsing lands.
  - No `User-Agent` at all — kills the demand-signal feedback loop for the CLI; we wouldn't know which install method to prioritise.
