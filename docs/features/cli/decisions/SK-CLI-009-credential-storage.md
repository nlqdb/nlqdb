# SK-CLI-009 — Credential storage: `zalando/go-keyring` primary, machine-keyed AES-GCM fallback file

- **Decision:** Long-lived credentials (CLI refresh tokens, MCP host keys) live in the OS keychain via `zalando/go-keyring`: Keychain (macOS), libsecret (Linux), Credential Manager (Windows). When the keychain is unavailable (some Linux distros without libsecret), the fallback is an AES-GCM-encrypted file at `~/.config/nlqdb/credentials.enc`. The AES-GCM key is HKDF(machine-id ‖ per-user salt) where the salt is a 32-byte random blob at `~/.config/nlqdb/.salt` (mode 0600) written on first use. A one-line warning fires at the moment the fallback is first written. **Plaintext is never an option.**
- **Core value:** Bullet-proof, Seamless auth
- **Why:** Keychain storage means credentials survive reboots, are encrypted at rest by the OS, and don't leak into shell history / `ps` output / env-dump screenshots. The AES-GCM fallback covers headless Linux without sacrificing encryption-at-rest. Mixing the per-user salt into HKDF is what keeps the file safe from another unprivileged process on the same box — `/etc/machine-id` is world-readable on Linux ([machine-id(5)](https://man7.org/linux/man-pages/man5/machine-id.5.html)), so machine-id alone wouldn't gate access. The salt's 0600 mode does. A backup of `credentials.enc` to a different host stays useless without that host's machine-id; a backup of `credentials.enc` + `.salt` together is the threat model the keychain path is for, not the fallback. This is the CLI manifestation of `GLOBAL-010`.
- **Consequence in code:** `cli/internal/credstore/` wraps `zalando/go-keyring` with the AES-GCM fallback. The fingerprint is SHA-256(machine-id) mixed with the per-user salt; the encryption key is HKDF-SHA256-Expand of that fingerprint. The salt file is created on first write with mode 0600 and reused thereafter (rotating it would un-decrypt every existing entry). The fallback path emits a single-line warning the first time it's written. No code path writes credentials to a plaintext file. Per `cli/AGENTS.md`'s `~/.config/nlqdb/` table, `.salt` is not a dotfiles-safe file.
- **Alternatives rejected:**
  - Plaintext `~/.nlqdb/credentials.json` — leaks via cloud backups / dotfile syncs.
  - GPG-encrypted file with a user passphrase — adds a passphrase prompt to every `nlq` invocation; fails the latency budget.
  - Use only the machine-id (no salt) as HKDF input — `/etc/machine-id` is world-readable, so any local unprivileged process could re-derive the key.
