# SK-CLI-015 — Background update check ≤ once/day; stderr only; auto-off in CI; explicit `nlq update` for curl-installed binaries

- **Decision:** The CLI checks `https://nlqdb.com/install/latest.json` at most once per 24 h (timestamped in `~/.config/nlqdb/state.json`), prints a one-line stderr notification when a newer version is available with the correct upgrade command for the user's install method, and is otherwise silent. The check auto-disables when `CI=true`, `GITHUB_ACTIONS=true`, `JENKINS_URL` is set, `--json` is passed, or stdout is non-TTY. `NLQDB_NO_UPDATE_CHECK=1` is the manual override. The explicit `nlq update` verb is curl-pipe-sh-only — it downloads the new signed tarball and atomically replaces `~/.local/bin/nlq`. brew / npm users see a notification telling them to run their package manager; `nlq update` is a no-op-with-hint on those install methods.
- **Core value:** Bullet-proof, Honest latency, Effortless UX
- **Why:** `gh`'s CI-noise problem is well-documented — every invocation in CI prints an update banner to stderr that pollutes logs and slows the call. CI auto-detection from day 1 means we don't ship that bug. Stderr (not stdout) keeps `--json` outputs clean for pipes per [`SK-CLI-004`](SK-CLI-004-human-output-default.md). Per-install-method upgrade copy means the notification is actionable: a brew user sees `brew upgrade nlqdb/tap/nlq`, not a curl URL that bypasses their package manager and leaves brew thinking nlq is still on the old version. The 24 h cadence balances "users learn about security patches" against "every invocation makes a network call." The `nlq update` verb gives curl-pipe-sh users a path that doesn't require re-running the install script and remembering the exact URL.
- **Consequence in code:** `cli/internal/updatecheck/` owns the once-per-day fetch, parses `latest.json` (`{"version": "0.1.6", "released_at": "...", "url_template": "...", "min_supported_version": "..."}`), and writes the timestamp + last-seen-version to `state.json`. The CI-detection helper (`isCi()`) is unit-tested against every env var listed in this decision. The `--json` flag short-circuits the check before it runs. `nlq update` is in `cli/cmd/nlq/update.go`; for non-curl install methods it prints the right hint and exits `0`. Reviewers reject any PR that prints update output to stdout, that runs the network call synchronously before the user's command executes (the fetch is post-response, fire-and-forget), or that bypasses the 24 h timestamp gate. The update endpoint is the only outbound HTTP call besides `api.nlqdb.com` — see [`SK-CLI-014`](SK-CLI-014-no-client-telemetry.md).
- **Alternatives rejected:**
  - No background check, manual `nlq update` only — users drift on stale versions for months; security patches don't propagate; same problem brew / npm have for users who never run their package manager's update verb.
  - Silent auto-update on next call — invasive (binary swap mid-session), security risk (the binary that just ran isn't the one on disk), inconsistent across install paths (brew users would have an auto-update fight with brew's own version pinning).
  - Check on every invocation — adds 50–200 ms to every call; `gh` learned this lesson the slow way and now batches.
