# SK-CLI-012 — Bare `nlq "<goal>"` mints from goal when no active DB; reuses the active DB otherwise

- **Decision:** Bare `nlq "<goal>"` resolves the DB context first: if `~/.config/nlqdb/state.json` carries an `active_db`, the goal asks against that DB; otherwise the CLI mints a new DB from the goal (same path as `nlq new "<goal>"`) and writes it as the active DB before asking. `nlq new "<goal>"` always creates a new DB and overwrites `active_db`. `nlq use <db>` switches the active DB without touching plans. The explicit form `nlq ask --db=<id> "<goal>"` skips active-DB resolution and is the only form that errors when the DB doesn't exist.
- **Core value:** Goal-first, Effortless UX, Seamless auth
- **Why:** [`SK-CLI-005`](SK-CLI-005-anonymous-first.md) promises bare invocations "immediately produce a working answer." Failing on a no-active-DB path on the very first call would flip the activation moment from "wow" to "homework" — the user just typed a goal and got a CLI error. Reusing the active DB after the first call matches the `§14.3` happy-path walkthrough (`nlq new "..."` followed by `nlq "..."` against the same DB). Routing `nlq new` and bare `nlq` through the same orchestration keeps `GLOBAL-017` honest — one way to ask, one way to create-and-ask, no third operation.
- **Consequence in code:** `cli/internal/state/` reads/writes `~/.config/nlqdb/state.json` (`active_db`, `last_used_at`, update-check timestamp). The bare-form handler in `cli/cmd/nlq/main.go` runs `resolveActiveDb()`, then either `askOnDb(activeDb, goal)` or `createDbFromGoal(goal) → setActiveDb(newDb) → askOnDb(newDb, goal)`. The state file is plaintext JSON (no secrets — DB ids are not credentials per `SK-APIKEYS-001`). `nlq new` writes `active_db` atomically; partial-write recovery is fine because the next `nlq new` overwrites it. Reviewers reject a `--no-create` flag on bare-form — `nlq ask --db=<id>` is the explicit form when the user wants a hard failure.
- **Alternatives rejected:**
  - Fail with "no active database — run `nlq new`" — better verb hygiene but trips users at the activation moment; contradicts `SK-CLI-005`.
  - Always mint a new DB on bare-form (ignore `active_db`) — every bare call balloons the user's DB list; the user's clear next-turn intent ("ask against the DB I just created") fails.
  - Resolve `active_db` from the most-recent `created_at` in the server-side DB list — adds a network call to every CLI invocation; offline / cache-hit paths can't fast-path; brittle if the user has many DBs.
