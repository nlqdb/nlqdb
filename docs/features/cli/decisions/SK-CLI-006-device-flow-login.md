# SK-CLI-006 — `nlq login` uses OAuth 2.0 Device Authorization Grant with `verification_uri_complete`

- **Decision:** `nlq login` runs the OAuth 2.0 Device Authorization Grant. The browser lands on `verification_uri_complete` (`/cli?code=<user_code>`) with the code pre-filled in the URL — one "Approve this device?" click, no typing. The raw user_code is printed as a fallback for SSH / headless / `--no-browser` cases. On approval the server mints a long-lived `sk_live_` key tied to the session user (via `mintSkLiveKey`, the same path the `/app/keys` dashboard uses); the CLI writes it to the OS keychain under `SlotRefreshToken` (per `SK-CLI-009`). Revocation is via the `/app/keys` dashboard (`SK-APIKEYS-011`).
- **Core value:** Seamless auth, Effortless UX, Bullet-proof
- **Why:** Device-code with `verification_uri_complete` is the lowest-friction sign-in for a CLI — no copy-paste of codes, no port-binding callback (which fails on remote SSH and behind firewalls), and the code is visible in the browser URL so the human verifies they're approving the right device. Issuing an `sk_live_` rather than a refresh+access JWT pair reuses `SK-APIKEYS-007`'s existing mint path and the CLI's existing keychain slot — no new JWT-issuer surface, no parallel revocation store. The trade-off (no 1h access-token rotation) is acceptable because the same key shape already secures every other backend integration; if rotation becomes necessary it lands as a single follow-up SK without re-plumbing the device flow.
- **Consequence in code:** `POST /v1/auth/device` returns `{device_code, user_code, verification_uri, verification_uri_complete, expires_in, interval}`; the CLI opens `verification_uri_complete` (or prints it on `--no-browser`) and polls `POST /v1/auth/device/token` at `interval` seconds. The approval page at `/cli` (session-gated; redirects to `/auth/sign-in?return_to=…` when anonymous) POSTs `/v1/auth/device/approve` with the `user_code` to mark the entry approved + stash the minted `sk_live_`. The token endpoint deletes the KV entry on read so a leaked `device_code` can't be replayed. KV TTL is 10 minutes; approval shortens the polling window to the remainder.
- **Alternatives rejected:**
  - localhost-callback OAuth — fails on SSH / headless; brittle behind firewalls.
  - Refresh + access JWT pair — would need a new JWT-issuer surface, a parallel revocation store, and per-call refresh logic in `cli/internal/api/`. The `sk_live_` path lands the device-flow UX without that surface area; revisit if rotation requirements appear.
  - Long-lived bearer tokens with no device flow — would force users into the `/app/keys` dashboard before the CLI is usable; breaks the goal-first single-command path.
  - Username/password — banned by `docs/architecture.md §4.1` ("No passwords, ever").
