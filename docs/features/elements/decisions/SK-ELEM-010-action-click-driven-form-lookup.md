# SK-ELEM-010 — `<nlq-action>` is click-driven; form association via DOM lookup, not form-associated CE

- **Decision:** `<nlq-action>` behaves like a submit-button — slot children render as the label; first click triggers the preview hop, second click on the rendered Apply button commits. Form data is collected from `this.closest("form")` by default; the optional `form="<id>"` attribute mirrors the native `<button form>` HTML attribute for forms outside the DOM ancestry. The element does **not** register as a form-associated custom element (no `static formAssociated = true`, no `attachInternals()`) — it doesn't participate in form submission as an input and has no own value to contribute.
- **Core value:** Simple, Effortless UX, Bullet-proof
- **Why:** Form-associated CEs (`attachInternals` / `setFormValue` / `formAssociatedCallback`) are designed for elements that submit *their own value* alongside other form fields. `<nlq-action>` aggregates other fields and dispatches the action; it has no value of its own. Walking up via `closest("form")` covers the common case (inside the form) and `form="<id>"` covers the rare case (outside the DOM ancestry) at zero bundle cost. Sticking to the form-element-not-form-control role also keeps the trust-UX gate (`SK-ELEM-012`) intelligible — the element owns the button affordance entirely instead of being one piece of a native submission pipeline.
- **Consequence in code:** `resolveForm()` is the only place form lookup happens — one branch (`form="<id>"` via `getElementById` on the root) plus the `closest("form")` fallback. No `ElementInternals` usage; the element doesn't appear in `form.elements`. Click events route through `handleClick`, which delegates to `preview()` / `applyDiff()` based on the current state (`SK-ELEM-012`).
- **Alternatives rejected:**
  - Form-associated custom element (`static formAssociated = true` + `attachInternals()`) — designed for elements that *contribute* a value; we don't, and the bundle cost (extra prototype + internals object) buys nothing the closest-ancestor lookup doesn't already give.
  - Intercept the parent form's `submit` event — race against any `<button type="submit">` the author leaves in place; surfaces a second affordance that's not the diff-preview button. v0.1 keeps the click-on-element trigger as the only path.
  - Pure attribute-only with no form lookup (author writes the column values inline) — defeats the "form-field-names inferred into columns automatically" pitch from `docs/features/elements/FEATURE.md`'s happy path.
