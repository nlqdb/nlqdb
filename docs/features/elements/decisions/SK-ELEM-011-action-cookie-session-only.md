# SK-ELEM-011 — `<nlq-action>` v0.1 supports cookie-session auth only; cross-origin write-token deferred

- **Decision:** The first slice of `<nlq-action>` authenticates through the same-origin Better Auth `__Host-session` cookie (works on `app.nlqdb.com` and self-host deployments). The `api-key=` attribute is honoured forward-compatibly — sent as `Authorization: Bearer …` — but `pk_live_*` is read-only per [`SK-APIKEYS-003`](../../api-keys/decisions/SK-APIKEYS-003-pk-live-readonly.md) and will 401 against the `/v1/ask` write path. The cross-origin "signed short-lived write-token" called out by `SK-APIKEYS-003` is **deferred to a follow-up slice** — see the open question in [`api-keys/FEATURE.md`](../../api-keys/FEATURE.md).
- **Core value:** Bullet-proof, Free, Effortless UX
- **Why:** Cookie-session writes already work today on the chat surface (`apps/web` is same-origin to `apps/api`). Designing the write-token shape (TTL, claim binding to `(account, dbId, verb)`, rotation) is its own decision and shouldn't block first-slice value: the diff-preview-then-confirm gate ([`SK-ELEM-012`](SK-ELEM-012-action-two-click-commit.md)) is auth-agnostic. Shipping with cookie-only is the smallest defensible first step — every embed currently authored on `app.nlqdb.com` works, every third-party embed is honestly rejected with `nlq-action:error` `kind="auth"` and a Sign-in CTA.
- **Consequence in code:** `fetchAsk` reuses the same `credentials: "include"` + optional Bearer header as `<nlq-data>`. `<nlq-action>` does not add a second auth path. The user-facing error message on a cross-origin 401 says "Sign in required to make changes" — clear next action per `GLOBAL-012`. The follow-up slice that lands the write-token will swap the auth header path without changing the element's public attribute surface.
- **Alternatives rejected:**
  - Ship a JWT-style write-token in v0.1 — locks in claim shape and TTL before any user has asked. The `api-keys/FEATURE.md` open question lists the unknowns; deciding now would violate `D1` ("resolve open questions first").
  - Use `sk_live_` from HTML — a single leaked HTML page becomes a full credential takeover; categorically rejected by `SK-APIKEYS-001`.
  - Extend `pk_live_*` semantics to writes — drops the structural read-only guarantee of [`SK-APIKEYS-003`](../../api-keys/decisions/SK-APIKEYS-003-pk-live-readonly.md); blast radius is much wider than fixing it later through write-tokens.
