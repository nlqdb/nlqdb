# SK-ELEM-014 — Render the `feature_gated` waitlist CTA inline; surface the structured body on `kind: "api"`

- **Decision:** When `POST /v1/ask` returns `403` with `error.status === "feature_gated"` ([`GLOBAL-027`](../../../decisions/GLOBAL-027-pre-alpha-gate.md)), `<nlq-data>` and `<nlq-action>` render an inline card containing the server's `message`, an `<a>` linking to `error.waitlist_url`, and the live BIRD / Spider lane progress from `error.gate`. `fetch.ts` no longer collapses 403 into `kind: "auth"`: only `401` does. `403` flows through the normal `kind: "api"` path, so the structured body reaches `render.ts` / `action-render.ts` (and `nlq-data:error` / `nlq-action:error` event listeners). The CTA card uses `data-kind="gated"`, `role="status"`, and `rel="noopener noreferrer"`. The `<nlq-action>` variant intentionally drops the retry button — the gate isn't transient, so retrying just 403s again. The `waitlist_url` is validated client-side: only `http:` / `https:` URLs render as a real link; anything else falls back to the generic `Error 403: feature_gated` message (defense-in-depth per OWASP XSS prevention).
- **Core value:** Onboarding (first-paint conversion when the gate trips) and Trust UX (no silent failure modes).
- **Why:** The end-to-end Hello World requires a `pk_live_*` key, and most new readers haven't been allowlisted yet, so the API returns `feature_gated` on their very first paste. The API painstakingly assembles a helpful body (waitlist URL, action text, live lane numbers); the element used to throw all of that away and render `"Authentication required."` — a message that's both wrong (the user has authenticated, the *feature* is gated) and dead-ends them. Rendering the CTA inline turns a confusing failure into a converting one without any extra effort from the embedder. Splitting 401 from 403 also generalises: structured 403 envelopes for other reasons (`account_required`, `pk_live_read_only`) likewise reach the renderer and show their server-provided slug instead of a generic auth message.
- **Consequence in code:** `packages/elements/src/fetch.ts` short-circuits only on `401` — `403` falls through to JSON parsing. `packages/elements/src/render.ts` exports `gatedBody` (narrows the failure to a `GatedBody` or `null`) and `gatedHtml` (renders the card); `errorHtml` calls them first and falls back to the existing message-based rendering. `action-render.ts` imports both helpers and emits the gated card in place of the retry button. Public event taxonomy stays inside the closed set from [`SK-ELEM-008`](SK-ELEM-008-events-and-aria-live.md) (`network` / `auth` / `api`); consumers branch on `detail.error.status === "feature_gated"` to react. URL validation rejects `javascript:` / `data:` / relative inputs.
- **Alternatives rejected:**
  - **Add a new `kind: "gated"` to the closed taxonomy ([`SK-ELEM-008`](SK-ELEM-008-events-and-aria-live.md)).** Wider blast radius (every consumer that switches on `kind` needs to learn the new value), no extra power vs. branching on `error.status` server-side, and the gate is one slug among many that `kind: "api"` already covers.
  - **Leave the renderer alone and ship better copy in the existing `kind: "auth"` message.** Doesn't help — the message still lacks the waitlist link and the live numbers, so users have to dig for the next step.
  - **Render the lane numbers as a progress bar.** Adds CSS the bundle can't ship in 6 KB ([`SK-ELEM-007`](SK-ELEM-007-bundle-size-cap.md)) and would compete with the embedder's own styling. Plain text keeps the element style-agnostic.
