# SK-LLM-032 — Schema-inference prompt requires insertable sample rows (FK-consistent, parent-first, NOT-NULL-complete)

Parent feature: [`llm-router/FEATURE.md`](../FEATURE.md). Extends the
`SCHEMA_INFER_SYSTEM` prompt (`packages/llm/src/prompts/schema-inference.ts`,
owned per [`SK-HDC-002`](../../hosted-db-create/decisions/SK-HDC-002-typed-schema-plan.md));
the deterministic floor under it is
[`SK-HDC-018`](../../hosted-db-create/decisions/SK-HDC-018-sample-insert-graceful-degradation.md).

- **Decision:** `SCHEMA_INFER_SYSTEM` gains a `sample_rows` validity contract: list a referenced table's rows **before** rows that reference it; every foreign-key value MUST equal a primary-key value present in an **earlier** sample row of the referenced table; every NOT NULL column MUST be present; and when unsure a row is valid, emit fewer rows rather than one that breaks a constraint. The shape spec is unchanged — this constrains the *values* the model fills in, not the schema grammar.
- **Core value:** Engine quality, Free
- **Why:** The compiler adds FK constraints (`ALTER ... ADD CONSTRAINT`) and NOT NULL before the provisioner inserts `sample_rows` in plan order ([`compile-ddl.ts`](../../../../apps/api/src/db-create/compile-ddl.ts), [`neon-provision.ts`](../../../../apps/api/src/db-create/neon-provision.ts)), so a seed row that references a not-yet-inserted parent, omits a NOT NULL column, or carries a bad type aborts the whole create with SQLSTATE class 22/23. The prompt previously said only "use realistic but small sample values" — zero insertability guidance — so the free chain at default temperature intermittently emitted unfittable rows; on 2026-06-08 this 500'd the FLOW-004 invited-stranger path 5/5 for "a meal planner for couples" (the schema had `meals.couple_id → couples.id` and `ingredients.meal_id → meals.id` FKs, fertile ground for an ordering/dangling-reference miss). Constraining the model up front is the cheapest, highest-leverage point to raise seed validity — pure scaffolding that compounds with the model ([`GLOBAL-026`](../../../decisions/GLOBAL-026-llm-strategy-byollm-hosted-premium.md)). It makes the *common* case a seeded first-value demo; `SK-HDC-018` is the guarantee that the *residual* failures still yield a working (un-seeded) DB rather than a 500.
- **Consequence in code:** `packages/llm/src/prompts/schema-inference.ts` appends five lines to the `SCHEMA_INFER_SYSTEM` array (≈55 input tokens per `schemaInfer` call — negligible against the schema-shape block already in the prompt). No plumbing change: every provider imports the one constant. The effect is probabilistic and observed on the deployed surface via the daily FLOW-004 walker's `first_value_quality` / `row_count` grade ([`SK-STRG-006`](../../stranger-test/FEATURE.md)) — a prompt change cannot be pinned by a deterministic unit test, which is exactly why the `SK-HDC-018` provisioner floor (unit-tested) carries the no-500 guarantee and this decision carries the seed-quality lift.
- **Alternatives rejected:**
  - **Prompt-only, no `SK-HDC-018` floor.** A probabilistic prompt can't guarantee insertable rows on every generation across every goal — without the deterministic retry, the residual still 500s the one funnel that reaches first-value. The two ship together.
  - **Lower the schema-infer temperature to reduce variance.** Trades seed-row variance for less schema creativity across the whole plan (table/metric design), and still gives no insertability guarantee. Out of proportion to the problem (P5).
  - **Drop `sample_rows` from the plan entirely (never seed).** Removes the failure but also removes the seeded first-value demo that makes the invited stranger's first query land on a populated table — the moat `SK-HDC-004` builds on the create moment. Degrade only on failure (SK-HDC-018), don't degrade by default.
